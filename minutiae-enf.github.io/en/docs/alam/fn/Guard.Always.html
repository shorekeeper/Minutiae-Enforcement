Guard::Always

built-in guard

A distinguished one-parameter guard value that ignores its event argument and returns true.

Static type(Event) -> Bool
Effectquery
ParametersExactly one event-compatible argument
ResultAlways true
Java representationValue.AlwaysV.INSTANCE
Allocation per useNone for the guard value
Runtime stateNone

Syntax

Guard::Always
always-guard =
    "Guard" "::" "Always" ;

Parameters

The source value declares no explicit parameter list. Its built-in function type accepts one argument assignable to the base Event parameter.

ParameterTypeUse
Event argument Any concrete event assignable to Event Ignored.

Constraints

  • The value may be used where a compatible one-parameter guard function is expected.
  • It is valid as the guard of an atomic or repeated single-event rule.
  • It is valid as a Step.guard because a step guard accepts one concrete event.
  • It is not assignable to a sequence rule guard requiring two or more event parameters.
  • The path is a value and must not be called with explicit parentheses in source.
Arity limitation

Guard::Always has one parameter. For a rule declaring several sequence events, write a lambda with the matching arity:

guard = (first, second) -> true;

Semantics

Application returns true without reading the event record or environment. It cannot fail due to a missing event field, unavailable cache entry, malformed regular expression, or arithmetic error.

Its static effect is query because it occupies the same function category as ordinary guards. The implementation itself performs no query.

Atomic and repeated rules

For Trigger::Atomic, every candidate event proceeds to the automaton throttle because the rule guard always passes.

For Trigger::Repeated, every candidate event enters the recurrence window. The repeated trigger evaluates the rule guard before recording each occurrence.

Sequence steps

As a step guard, the value admits every incoming event whose event kind matches that step. It does not affect sequence partitioning, expiry, re-anchoring, or the final rule guard.

No bypass

An always-true guard bypasses only conditional filtering. Trigger state, sequence order, time windows, automaton throttle, self-muting, verdict evaluation, sanction resolution, permissions, and dry-run posture remain in effect.

Effect

query by static function type. The value is assignable to compatible query guard positions and performs no actual environment read.

Implementation notes, non-normative

The elaborator recognizes the path specially:

if (p.segments().size() == 2
        && p.segments().get(0).equals("Guard")
        && p.segments().get(1).equals("Always")) {
    tables.refOf.put(
        p,
        Symbol.AlwaysSym.INSTANCE
    );

    return new Func(
        List.of(Builtins.ANY_EVENT),
        BOOL,
        Effect.QUERY
    );
}

Compile-time and runtime path evaluation return one singleton value:

case Symbol.AlwaysSym ignored ->
    Value.AlwaysV.INSTANCE;

Application ignores the supplied argument list:

case Value.AlwaysV ignored ->
    new Value.BoolV(true);
  • Arity correctness is established statically by function assignability.
  • The runtime application branch does not inspect argument count.
  • A new Value.BoolV(true) is produced for each application.
  • The guard itself captures no lexical environment.
  • No cache, storage, Bukkit, or Paper API is consulted.

Example

rule<Break> every_break {
    condition {
        trigger = Trigger::Atomic;
        guard = Guard::Always;
    }
    => verdict =
        break_verdict<Measure::CENSURE>;
}

As a sequence step guard:

Step<Login> {
    .guard = Guard::Always,
}
