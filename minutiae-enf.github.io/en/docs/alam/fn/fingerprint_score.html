fingerprint_score

function

Returns the currently cached best-match probability produced by the fingerprint evasion model for one online account.

Signature(Account) -> Real
Effectquery
Conventional range0 through 1
Cache lifetime30 seconds
Miss result0.0
Offline result0.0, cached for the TTL
Blocking I/ONone on the guard evaluation path

Syntax

fingerprint_score(subject)
fingerprint-score-call =
    "fingerprint_score" "("
        account-expression
    ")" ;

Parameters

ParameterTypeDescription
subject Account Account whose currently available connection probe is scored against stored signals.

Constraints

  • The call requires exactly one Account argument.
  • The call is valid only under a query effect allowance.
  • The live fill path requires the subject to resolve to an online Paper player.
  • An unknown account returns 0.0.
  • The function does not block while a score is being computed.

Semantics

The function reads a per-account cache. Its result is the probability of the best matching sanctioned account found by the current fingerprint model, or 0.0 when no match is available.

Cache behavior

Cache stateImmediate resultBackground action
FreshCached scoreNone
Missing0.0Schedule scoring when the player is online
ExpiredStale cached scoreSchedule refresh
Offline during fillCurrent neutral or stale resultPublish 0.0 for 30 seconds
Scoring failureCurrent neutral or stale resultPublish 0.0 for 30 seconds
Guard design

The first evaluation for an uncached subject normally observes 0.0. Do not assume that a high stored match is synchronously available on the event that initiated the cache fill.

Score source

The fill path calls the fingerprint service for the currently online player. It constructs a probe from the player's current address and name, queries matching signals of active connection-blocking sanctions, and selects the best candidate under the evidence model.

The resulting value is a posterior match probability, not the legacy sum of configured capture weights. It can include the bounded behavioral session-correlation adjustment.

Distinction from Evasion.score

Evasion.score is a snapshot supplied by the event producer. fingerprint_score reads a mutable cache and may return a newer, older, stale, or neutral value.

Disabled fingerprint subsystem

The live guard environment still calls the injected fingerprint service. When fingerprint detection is disabled, ordinary capture and pre-login assessment paths are disabled, but this function's exact result depends on the service and stored data available to its scoring call. Definitions should treat 0.0 as unavailable or no match.

Effect

query The function reads live online-player state and a mutable asynchronous score cache.

It may appear in rule and sequence step guards. It cannot appear in verdicts, transforms, constants, matrix cells, trigger construction, or template constraints.

Implementation notes, non-normative

The signature is registered as:

"fingerprint_score", new Func(
    List.of(ACCOUNT),
    REAL,
    Effect.QUERY
)

The interpreter delegates directly:

case "fingerprint_score" ->
    new Value.RealV(
        f.env.fingerprintScore(
            accountOf(args.get(0))
        )
    );

The live environment performs a non-blocking lookup:

final Entry entry =
    scoreCache.get(subject);

if (entry == null
        || entry.expiresAt() <= now) {
    fillScore(subject, now);
}

return entry == null
    ? 0.0d
    : entry.value();

An offline subject is cached as neutral:

final Player player =
    plugin.getServer().getPlayer(subject);

if (player == null) {
    scoreCache.put(
        subject,
        new Entry(0.0d, now + TTL_MS)
    );
    scorePending.remove(subject);
    return;
}

The asynchronous completion publishes the probability:

fingerprint.scorePlayer(player)
    .whenComplete((match, error) -> {
        final double score =
            error != null || match == null
                ? 0.0d
                : match.score();

        scoreCache.put(
            subject,
            new Entry(score, now + TTL_MS)
        );

        scorePending.remove(subject);
    });
  • The cache key is the subject UUID.
  • A concurrent pending set permits one fill per subject at a time.
  • The TTL begins at fill request time, not completion time.
  • scorePlayer disables reverse-DNS derivation for this diagnostic scoring path.
  • The cache is not immediately invalidated when fingerprint evidence changes.

Example

rule<Login> probable_evasion {
    condition {
        trigger = Trigger::Atomic;
        guard = (event) ->
            fingerprint_score(event.subject) >= 0.90;
    }
    => verdict =
        evasion_verdict<Measure::SUSPENSION, 30d>;
}
