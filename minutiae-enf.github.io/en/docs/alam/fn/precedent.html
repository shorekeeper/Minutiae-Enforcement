precedent

function

Returns the cached count of prior non-warning sanctions for one account under one primary rule.

Signature(Account, Rule) -> Int
Effectquery
Live sourceBounded in-memory cache backed by asynchronous storage queries
Cache keyAccount UUID and rule identifier
Cache lifetime30 seconds
Miss result0
Blocking I/ONone on the guard evaluation path

Syntax

precedent(subject, rule)
precedent-call =
    "precedent" "("
        account-expression ","
        rule-identifier-expression
    ")" ;

Parameters

ParameterTypeDescription
subject Account Account whose rule-scoped prior record is queried.
rule Rule Primary external rule identifier defining the precedent scope.

Constraints

  • The call requires exactly two arguments.
  • The first argument must have nominal type Account.
  • The second argument must have primitive type Rule, not structural Rule<Events...>.
  • The call is permitted only under a query effect allowance.
  • Every rule identifier literal reachable from a published guard must exist in the external rule registry.
  • An unknown account representation or null rule produces the neutral result 0.

Semantics

The returned value is the number of eligible prior sanction rows whose effective measure is not Measure::WARN. It is not the count of all sanctions and does not include the separate formal-warning count.

A row contributes only when:

  • its subject UUID equals subject;
  • its primary rule equals rule;
  • it has not been vacated, or it was lifted under a kind that retains precedent;
  • its decay deadline is zero or remains in the future.
Returned information

The function exposes priorSanctions only. It does not expose prior warnings, active probation, lift kind, sanction activity, expiry, or joinder-chain precedent.

Cache behavior

A fresh cache entry returns its stored count immediately. A missing entry returns 0 and schedules one asynchronous storage query. Repeated misses for the same key while that query is pending do not schedule duplicate queries.

An expired entry returns its stale value immediately and schedules a refresh. The stale value remains available until the asynchronous query publishes a replacement.

FreshCached countNone
Missing0Schedule fill
ExpiredStale cached countSchedule refresh
Query failureCurrent cached or neutral valueRemove pending marker; no replacement published
First-observation behavior

A guard may evaluate false on the first event for an uncached account-rule pair even when storage contains precedent. A later event can observe the asynchronously filled value.

Rule scope and joinder

The built-in function uses ordinary subject-and-rule precedent. It does not call the storage operation that aggregates a woven joinder chain. Automatic sanction execution may later use chain precedent when the materialized command contains a weave parent, but this function does not expose that calculation to guards.

Activity and expiry

The precedent query does not require a sanction to remain active or unexpired. A completed or expired sanction may continue to count until it is vacated or decays out of precedent.

Effect

query The function reads mutable enforcement state through GuardEnvironment.

It may be called in rule guards and sequence step guards. Calling it in a constant initializer, matrix cell, transform, trigger expression, template constraint, or verdict produces E007.

Implementation notes, non-normative

The built-in signature is registered as:

"precedent", new Func(
    List.of(ACCOUNT, RULE),
    INT,
    Effect.QUERY
)

The interpreter decodes the account and delegates to the environment:

case "precedent" -> {
    final UUID subject =
        accountOf(args.get(0));

    final String rule =
        ((Value.RuleIdV) args.get(1)).id();

    yield new Value.IntV(
        f.env.precedent(subject, rule)
    );
}

The live environment performs non-blocking cache access:

final Entry entry =
    precedentCache.get(key);

if (entry == null
        || entry.expiresAt() <= now) {
    fillPrecedent(
        key,
        subject,
        rule,
        now
    );
}

return entry == null
    ? 0L
    : (long) entry.value();

One pending marker suppresses duplicate fills:

if (!precedentPending.add(key)) {
    return;
}

storage.precedent(
    subject.toString(),
    rule,
    now
).whenComplete((value, error) -> {
    if (error == null && value != null) {
        precedentCache.put(
            key,
            new Entry(
                value.priorSanctions(),
                now + TTL_MS
            )
        );
    }

    precedentPending.remove(key);
});

The storage aggregate separates non-warning sanctions from warnings:

SELECT
    SUM(
        CASE
            WHEN measure = 'WARN' THEN 0
            ELSE 1
        END
    ) AS reals,
    SUM(
        CASE
            WHEN measure = 'WARN' THEN 1
            ELSE 0
        END
    ) AS warns
FROM bans
WHERE uuid = ?
  AND rule = ?
  AND (lifted_at = 0 OR lift_kind <> 0)
  AND (decay_at = 0 OR decay_at > ?)
  • The cache stores the integer count in a double-valued entry shared with score caching, then casts it back to long.
  • The expiry timestamp is based on the time at which the fill was requested.
  • Cache entries are not explicitly invalidated immediately after sanction issuance or lifting.
  • The 30-second TTL bounds eventual staleness under ordinary operation.

Example

rule<Chat> repeat_offender {
    condition {
        trigger = Trigger::Atomic;
        guard = (event) ->
            precedent(event.subject, P.2.1) >= 2;
    }
    => verdict =
        repeated_flood_verdict<Measure::MUTE, 1d>;
}
