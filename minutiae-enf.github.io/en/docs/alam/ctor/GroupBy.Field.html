GroupBy::Field

constructor

Partitions stateful trigger data by the textual rendering of one named field in the incoming raw event facts.

Fields

FieldTypeRequiredDescription
name Text Yes Name used to retrieve a raw fact from every incoming event.

Constraints

  • The constructor requires exactly one field name.
  • The field name value must have type Text.
  • The compiler does not currently verify that the named field exists on the containing rule's event type.
  • For a sequence, the selected field should exist and remain stable across every step event intended to share one partial match.
  • The source field name is case-sensitive.

Semantics

For each event, the runtime:

  1. Reads the raw fact map by the exact source name.
  2. Converts the resulting Java object with String.valueOf.
  3. Combines the rendered text with automaton and rule names.
  4. Uses the resulting string as the key of WindowTracker or SequenceTracker.

Values compare by their rendered Java text rather than by ALAM type identity. The following raw values produce separate partitions:

Long.valueOf(1)       -> "1"
Double.valueOf(1.0)   -> "1.0"
Boolean.TRUE          -> "true"
"1"                   -> "1"

A numeric long value 1 and text value "1" collide because both render as 1.

Missing fields

EventFacts.field(name) returns Java null when the field is absent. String.valueOf(null) yields the literal partition text null.

All events missing the field therefore share one partition. An actual text value equal to "null" shares that same partition.

Operational requirement for sequences

A later step can advance an earlier partial only when it derives the identical complete partition key. A field present on the first event but absent on the second event normally prevents completion.

Raw facts versus typed event fields

Partition lookup uses the raw EventFacts map, not the adapted ALAM record. The neutral-value conversion performed by EventAdapter does not apply.

For example, an absent integer event field appears as 0 inside a guard after adaptation, but GroupBy::Field sees the missing raw fact and uses null as the partition value.

Implementation notes, non-normative

The constructor lowers to a field-kind partition record:

new TriggerPlan.Part(
    TriggerPlan.Part.Kind.FIELD,
    ((Value.TextV) groupBy.field(0)).value()
)

Runtime key construction is direct:

case FIELD ->
    String.valueOf(facts.field(part.field()));

The complete key is:

return rule.automaton()
    + '|'
    + rule.plan().name()
    + '|'
    + partition;
  • The separator does not need escaping because the key is opaque and never parsed.
  • The field value's Java toString behavior becomes part of the partition identity.
  • Current live event adapters use simple scalar facts, so unstable object identity strings are not expected.
  • Simulation constructs the same EventFacts shape as live dispatch and therefore uses identical partition rendering when state is consulted.

Example

rule<Break> repeated_material {
    condition {
        trigger = Trigger::Repeated {
            .count = 8,
            .within = 1m,
            .group_by = GroupBy::Field("block"),
        };
        guard = Guard::Always;
    }
    => verdict =
        break_verdict<Measure::CENSURE>;
}

Each block material accrues an independent recurrence window. Eight stone breaks do not advance the dirt partition.
