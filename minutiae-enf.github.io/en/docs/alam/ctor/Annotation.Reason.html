Constraints

  • The constructor requires exactly one Text value.
  • The constructor is command-scoped and produces R005 in a stamped Layout.
  • The system actor must hold minutiae.annotation.reason after materialization.
  • The compiler does not impose a maximum reason length.
  • An empty text value is type-correct but does not suppress downstream fallback behavior.

Semantics

Unlike most annotation constructors, Annotation::Reason does not materialize to @reason(text). It writes the command scalar override:

overrides["reason"] = text

The resolver selects reason text in this order:

  1. non-empty reason= scalar override;
  2. effective command annotation @reason(text);
  3. selected layout reason;
  4. description of the primary rule;
  5. no reason.

Because ALAM materialization uses the scalar override, a non-empty Annotation::Reason value takes precedence over layout and rule fallbacks.

Empty-text behavior

An empty scalar override is ignored by resolveReason. In an ALAM-produced sanction with no selected layout and no effective command @reason token, the resolver falls back to the primary rule description.

Persistence and display

The resolved reason is persisted as plain text. It is not localized after storage and is not interpreted as message markup.

Player-facing and staff-facing surfaces may abbreviate the visible inline text while retaining the full reason in a hover card. The stored reason remains unchanged.

Effective measure distinction

A warn-first downgrade changes the effective measure but does not rewrite the reason. The same reason text is associated with the formal warning.

Missing reason at effect time

If the resolved reason is null, player-facing enforcement code may substitute the localized display name of the effective measure. Persisted fallback strings used by behavioral state remain canonical measure names because stored text cannot be localized per future viewer.

Implementation notes, non-normative

Materialization writes the scalar override directly:

case "Reason" ->
    overrides.put("reason", text(a, 0));

Resolver precedence is explicit:

final String override =
    overrides.get("reason");

if (override != null && !override.isEmpty()) {
    return override;
}

final RawAnnotation reasonAnn =
    effective.get("reason");

if (reasonAnn != null
        && !reasonAnn.positional().isEmpty()) {
    return reasonAnn.positional().get(0);
}

if (layout != null
        && layout.reason() != null) {
    return layout.reason();
}

return rule != null
    ? rules.describe(rule)
    : null;
  • The reason is written to SanctionRow.reason.
  • The message service sanitizes scalar placeholder values so reason text cannot inject legacy color codes.
  • The web panel and audit projections display persisted text as stored.
  • The reason string is included in public broadcast and notification templates when those paths are enabled.
