Trigger::Sequence

constructor

A stateful trigger that admits guarded event steps in declaration order, captures their facts, and fires when the complete sequence occurs within one partition and time span.

ContainerTrigger
CategoryStateful trigger constructor
Observed eventsTwo or more, one per step
Compile-time effectpure
Step-guard effectquery
Runtime stateOne partial match and fact-capture list per active partition
Completion behaviorDrain capture, evaluate rule guard, then acquire automaton throttle

Syntax

Trigger::Sequence(within, group_by, steps)                  (1)

Trigger::Sequence {
    .within = within,
    .group_by = group_by,
    .steps = steps,
}                                                            (2)
  • (1)Positional construction.
  • (2)Named-field construction.

Fields

FieldTypeRequiredDescription
within Duration Yes Maximum elapsed time from admission of the first step to admission of the final step.
group_by GroupBy Yes Selector producing the independent sequence partition.
steps Step[] Yes Ordered event kinds and admission guards.

Constraints

Ill-formed if
  1. within is zero, negative, or permanent.
  2. The steps list contains fewer than two elements.
  3. The number of steps differs from the number of events declared by the containing rule.
  4. The event type of step i differs from declared event i.
  5. A field is absent, duplicated, unknown, or has the wrong type.
  6. The positional constructor receives other than three arguments.

Each step guard must accept one event of its step's event type and return Bool. A step guard may call query-capable built-ins.

The compiler checks event correspondence positionally. It does not sort steps or infer the containing rule's event list from them.

Semantics

Each partition holds one running partial match:

StateMeaning
stepZero-based index of the next expected step.
startEpoch-millisecond timestamp at which the current partial admitted its first step.
Captured factsEvents admitted for completed steps, retained in order.

Advancement

For an incoming event in partition P:

  1. Load the partial match for P, creating an empty partial when none exists.
  2. If a partial is active and now - start > within, reset its expected step to zero.
  3. Compare the event kind with the next expected step kind.
  4. If the kinds match, adapt the event to that step's record shape and evaluate the step guard.
  5. If the guard returns true, admit and capture the event.
  6. If this was the first step, set start = now.
  7. If this was the final step, reset the tracker, drain captured facts, and proceed to the rule guard.
Span boundary

A partial is stale when:

now - start > within

A final event exactly within milliseconds after the first event remains eligible. This differs from the strict lower-bound rule used by Trigger::Repeated.

Non-matching events

An event that does not satisfy the next expected step normally leaves the partial unchanged. The sequence does not require participating events to be adjacent in the global event stream. Unrelated events may occur between admitted steps.

Re-anchoring

When a later step is expected and an incoming event instead satisfies step zero, the partial match restarts from that event:

start = now
step = 1

Re-anchoring supports overlapping starts. A new valid first step replaces an older partial rather than being discarded merely because the tracker was waiting for another step.

Fact capture

Every admitted step contributes one EventFacts snapshot. On completion, the runtime adapts captured facts to the parallel event-record shapes in the rule plan.

The resulting argument vector is passed to both the rule guard and verdict:

guard(event1, event2, ..., eventN)
verdict(event1, event2, ..., eventN)

The completing event supplies the final argument. Earlier arguments come from the partition's capture list.

Postcondition on completion

The sequence tracker is reset and the capture list is removed before the rule guard result is known. If the rule guard returns false, the completed sequence is still consumed and must be accrued again.

Failure behavior

A step-guard interpreter failure is treated as a non-match. A rule-guard failure after completion is treated as an unsatisfied guard. A verdict failure suppresses sanction issuance and writes an English operator warning.

Simulation

Simulation does not consult or mutate a live partial match. A sequence candidate passes trigger simulation only when the synthetic event has the final step's event kind and satisfies the final step guard.

Because earlier events are unavailable, the simulator adapts the same synthetic completing facts under every event shape in the argument vector. Simulation therefore verifies final-step and rule behavior but is not an exact rehearsal of captured cross-event values.

Implementation notes, non-normative

Lowering converts each Step value to a compact runtime entry:

steps[i] = new TriggerPlan.Sequence.Step(
    kindOf(step.event().name()),
    step.guard()
);

The tracker stores mutable progress objects in a concurrent map:

private static final class Progress {
    int step;
    long start;
}

private final Map<String, Progress> partitions =
    new ConcurrentHashMap<>(64);

The core transition is:

if (p.step > 0 && now - p.start > withinMs) {
    p.step = 0;
}

if (matcher.matches(p.step, facts)) {
    if (p.step == 0) {
        p.start = now;
    }

    p.step++;

    if (p.step == stepCount) {
        p.step = 0;
        return true;
    }

    return false;
}

if (p.step > 0 && matcher.matches(0, facts)) {
    p.start = now;
    p.step = 1;
}

DispatchEngine maintains capture lists separately from tracker progress. It observes the step index before and after advance to decide whether to append, replace, or drain facts.

  • Tracker partitions are swept every 1024 advances.
  • An empty partial or a partial older than one hour is eligible for eviction.
  • Capture entries are swept every 512 sequence gate operations when the corresponding tracker step is zero.
  • The partition map is concurrent, but each Progress object is mutable and not internally synchronized. Correct operation assumes that one partition is not advanced concurrently from multiple dispatch threads.
  • Live Paper chat events may enter dispatch from an asynchronous thread, while login and break events generally enter from the main thread. Definitions that combine such event kinds into one partition should be reviewed with this execution model in mind.
Implementation precision

The tracker correctly re-anchors its numeric progress. Capture synchronization is maintained in a separate structure by comparing before and after indexes. Concurrent advancement of one partition is not serialized by the tracker itself.

Complexity
Step attempt
O(Gs), where Gs is the evaluated step-guard cost. A re-anchoring attempt may evaluate both the expected step and step zero.
Completion
O(N + Gr + V), where N is step count, Gr rule-guard cost, and V verdict cost.
Storage
O(N) captured events per active partition, plus constant tracker progress.
Tracker sweep
O(P) every 1024 advances, where P is tracked partition count.
Capture sweep
O(C) every 512 sequence gates, where C is capture partition count.

Example

rule<Login, Chat, Break> staged_evasion {
    condition {
        trigger = Trigger::Sequence {
            .within = 15m,
            .group_by = GroupBy::Subject,
            .steps = {
                Step<Login> {
                    .guard = (login) ->
                        fingerprint_score(login.subject) >= 0.80,
                },
                Step<Chat> {
                    .guard = (chat) ->
                        chat.length > 120,
                },
                Step<Break> {
                    .guard = (broken) ->
                        broken.block == "CHEST",
                },
            },
        };
        guard = (login, chat, broken) ->
            login.subject == chat.subject
            && chat.subject == broken.subject;
    }
    => verdict =
        evasion_verdict<Measure::SUSPENSION, 7d>;
}
