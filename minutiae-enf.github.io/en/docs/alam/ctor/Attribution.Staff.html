Attribution::Staff

constructor

Requests that an automatically produced sanction be recorded under one explicit staff display name.

Fields

FieldTypeRequiredDescription
name Text Yes Display name persisted as the sanction's attributed staff identity.

Constraints

  • The constructor requires exactly one Text value.
  • The system actor must hold minutiae.annotation.as.
  • The name is not resolved against online players, account history, a staff registry, or rank configuration.
  • The current validator requires one positional value but does not require non-empty text.
  • Attribution does not grant the named staff member's permissions to automation.
  • Attribution does not change the command sender that receives success and failure reports.

Semantics

The constructor materializes to:

@as(name)

The shared resolver checks the system actor's permission and stores the supplied text as ResolvedSanction.attributedStaff.

At execution, the persisted staff name is selected as:

staff =
    attributedStaff != null
        ? attributedStaff
        : issuer.getName()

Observable consequences

The explicit name affects more than display. It becomes the authoritative staff string used by several subsystems:

  • the sanction row;
  • issuer lines in reports;
  • public sanction broadcasts;
  • notification summaries;
  • staff-based case queries;
  • bulk-lift criteria by staff name;
  • foreign-lift ownership checks.
Ownership consequence

A later sender whose name differs from the stored attributed name is treated as lifting another staff member's sanction and requires minutiae.lift.foreign.

Authority remains with the system actor

Permission checks continue to use the configured system actor predicate. Naming "Administrator" does not grant administrator permissions, and naming a real player does not consult that player's effective permission set.

Rank display distinction

The rank identifier used by broadcasts is resolved from the actual issuing CommandSender. The attributed name and displayed rank may therefore refer to different conceptual identities.

For automatic dispatch, the issuer is normally the console sender while the stored staff name is the supplied constructor text.

Empty name

An empty Text payload is accepted by current structural and command parameter validation. Because the value is non-null, the executor persists the empty string rather than falling back to the issuer name.

Recommendation

Use a non-empty stable attribution name. The compiler does not currently enforce this operational requirement.

Implementation notes, non-normative

The constructor is represented as one text field:

new Ctor(
    "Staff",
    List.of(new Field("name", TEXT))
)

Materialization appends the command annotation immediately:

if (attribution.ctor().equals("Staff")) {
    annotations.add(directive(
        "as",
        ((Value.TextV) attribution.field(0)).value()
    ));
} else {
    systemAttribution = true;
}

The resolver extracts the first positional parameter:

final String attributedStaff =
    firstPositional(effective, "as");

The executor persists the selected name:

final String staff =
    r.attributedStaff() != null
        ? r.attributedStaff()
        : issuer.getName();
  • The annotation permission follows the catalog convention minutiae.annotation.as.
  • The attributed name is plain text and is not a UUID.
  • Rank resolution happens separately through RankConfig.resolve(issuer).
  • Audit actions performed later record their own acting sender rather than automatically reusing the sanction attribution.
  • The web and command surfaces may impose different bounded permission predicates, but both use the same resolver.
