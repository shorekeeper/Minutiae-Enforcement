Annotation::Decay

constructor

Defines the interval after which a persisted sanction stops contributing to future precedent calculations.

Fields

FieldTypeRequiredDescription
window Duration Yes Interval from effective issue time to removal from precedent.

Constraints

  • The constructor requires exactly one primitive Duration.
  • The ALAM primitive cannot represent the permanent duration term.
  • A zero span is type-correct and means that the sanction leaves precedent at its issue timestamp.
  • The system actor must hold minutiae.annotation.decay.
  • The constructor does not alter the sanction's active duration or expiry.

Semantics

At sanction execution:

decay_at =
    issued_at + window

issued_at is the effective issue timestamp after any backdate offset. Decay therefore follows the sanction's legal timeline rather than the wall-clock time at which the row was inserted.

Future precedent queries include the sanction only while:

decay_at == 0
|| decay_at > now

A finite decay constructor always produces a non-default timestamp unless arithmetic yields zero from an epoch-zero edge case. A sanction without this annotation stores decay_at = 0, meaning that it never decays automatically.

Effect boundary

Decay changes only future precedent counts. It does not lift the sanction, clear active behavior, shorten expiry, remove the row from the docket, erase audit history, or delete evidence.

Warnings and non-warning sanctions

Decay filtering applies to both warning and non-warning rows. Once the timestamp passes:

  • the row no longer contributes to priorWarnings;
  • the row no longer contributes to priorSanctions;
  • the row may still remain active if its enforcement expiry lies later;
  • the row remains visible unless another lifecycle rule hides or deletes it.

Interaction with lifts

Vacated sanctions are excluded from precedent immediately, regardless of decay. Pardoned and time-served sanctions remain eligible until their decay timestamp passes.

Implementation notes, non-normative

Materialization creates the canonical command token:

case "Decay" ->
    out.add(directive("decay", dur(a)));

The executor computes the persisted timestamp:

final long decayAt =
    r.decayFor() != null
        ? issuedAt + r.decayFor().millis()
        : 0L;

Ordinary precedent uses an indexed storage predicate:

WHERE uuid = ?
  AND rule = ?
  AND (lifted_at = 0 OR lift_kind <> 0)
  AND (decay_at = 0 OR decay_at > ?)
  • The database has an index on (uuid, rule, decay_at).
  • No maintenance task rewrites a row when decay occurs.
  • Decay is evaluated lazily each time precedent is queried.
  • The current timestamp arithmetic uses ordinary Java addition rather than Math.addExact at sanction execution.

Example

.annotations = {
    Annotation::WarnFirst(2),
    Annotation::Decay(90d),
},

The warning or sanction remains recorded, but stops affecting warning and escalation counts 90 days after its effective issue time.
