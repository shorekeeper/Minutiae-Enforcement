Annotation::WarnFirst

constructor

Defines how many prior formal warnings a subject must accumulate under the primary rule before the selected enforcement measure is applied.

ContainerAnnotation
CategoryPrecedent gate constructor
PayloadOne Int quota
Layout scopePermitted
Command form@warn-first(count)
Effective downgradeMeasure::WARN
Runtime stateNone in memory; prior warnings are read from storage

Syntax

Annotation::WarnFirst(count)                               (1)

Annotation::WarnFirst {
    .count = count,
}                                                           (2)
  • (1)Positional construction.
  • (2)Named-field construction.

Fields

FieldTypeRequiredDescription
count Int Yes Positive warning quota checked against prior warnings under the primary rule.

Constraints

  • The constructor requires one Int value.
  • After materialization, the value must fit a positive Java 32-bit integer.
  • A sanction without a primary rule has no useful warning-precedent scope.
  • The system actor must hold minutiae.annotation.warn-first.
  • The constructor is valid in layouts and direct sanction descriptors.
Downstream numeric validation

The ALAM type accepts any signed 64-bit Int. Positivity and Java int range are enforced after materialization by the command annotation validator.

Semantics

Let:

Q = configured warning quota
W = prior warning count under the primary rule

The offense is downgraded when:

Q > 0
&& !now_override
&& resolved_measure != Measure::WARN
&& W < Q

When downgraded:

  • the effective measure becomes Measure::WARN;
  • the behavioral mask becomes zero;
  • the sanction is persisted as a warning;
  • the warning contributes to future warning count;
  • the originally selected measure is not applied;
  • escalation and temporal duration do not create an active restriction for this offense.

Once W >= Q, the originally resolved measure proceeds normally. The warning quota does not reset after the first real sanction.

Immediate bypass

The command-side @now annotation bypasses warn-first for one sanction. ALAM has no corresponding annotation constructor, but automation may still materialize or receive it through another command-surface mechanism.

Precedent scope

Warnings are counted separately from non-warning sanctions. Vacated warning records are excluded from precedent. Warnings whose decay time has passed are also excluded.

For a sanction woven into a case chain, execution may consult chain precedent instead of ordinary subject-and-rule precedent.

Implementation notes, non-normative

Materialization converts the ALAM long value to decimal text:

case "WarnFirst" ->
    out.add(directive(
        "warn-first",
        Long.toString(
            ((Value.IntV) a.field(0)).value()
        )
    ));

The command validator then parses a Java int:

final int value;
try {
    value = Integer.parseInt(
        a.positional().get(0)
    );
} catch (final NumberFormatException e) {
    return "not an integer: '"
        + a.positional().get(0)
        + "'";
}

return value > 0
    ? null
    : "must be a positive integer";

The execution gate is direct:

return r.warnFirst() > 0
    && !r.now()
    && r.measure() != Measure.WARN
    && precedent.priorWarnings()
        < r.warnFirst();
  • The executor computes the gate after asynchronous precedent lookup.
  • A decision trace records prior warnings and whether downgrade occurred.
  • The persisted warning number is priorWarnings + 1.
  • No behavioral or connection-blocking propagation occurs for a downgraded warning.
Complexity
Gate evaluation
O(1) after precedent has been loaded.
Precedent lookup
Asynchronous indexed storage query.
Stored state
No separate quota state; warnings are ordinary sanction rows counted by query.

Example

.annotations = {
    Annotation::WarnFirst(2),
    Annotation::Notify("staff"),
},

The first two applicable offenses become warnings. The third and later offenses apply the descriptor's selected measure.
