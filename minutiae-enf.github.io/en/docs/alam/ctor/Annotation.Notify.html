Annotation::Notify

constructor

Requests delivery of a sanction summary to one configured notification channel.

ContainerAnnotation
CategoryPolicy annotation constructor
PayloadOne Text channel name
Layout scopePermitted
Command form@notify(channel)
Delivery timeAfter successful sanction persistence and application

Syntax

Annotation::Notify(channel)                                (1)

Annotation::Notify {
    .channel = channel,
}                                                           (2)
  • (1)Positional construction.
  • (2)Named-field construction.

Fields

FieldTypeRequiredDescription
channel Text Yes Name of a channel declared under notify.channels in runtime configuration.

Constraints

  • The constructor requires exactly one Text value.
  • The ALAM compiler does not verify that the named channel exists in runtime configuration.
  • Channel lookup is case-insensitive after sanction resolution because resolved names are normalized to lowercase.
  • The system actor must hold minutiae.annotation.notify.
  • The constructor is valid in both Layout.annotations and Sanction.annotations.

Semantics

The constructor materializes to one command annotation:

@notify(channel)

The shared sanction resolver validates the token, checks the system actor's permission, and stores the lowercase channel name in the resolved sanction.

After a live sanction completes, the executor asks NotificationService to render the sanction summary and deliver it through the selected channel.

Channel typeDelivery behavior
STAFFRenders per online recipient and requires the channel's configured permission.
CONSOLERenders for and sends to the server console on the main thread.
LOGSerializes the default-locale message to plain text and writes it to the plugin log.
WEBHOOKPosts a plain-text JSON payload through the asynchronous scheduler.
Unknown channel behavior

An undefined channel does not reject the sanction. The notification service logs the unknown channel once and skips delivery.

Multiple notification values

The command annotation grammar supports several positional channels in one token. The ALAM constructor stores one channel only.

Constructing several Annotation::Notify values does not reliably create one multi-channel request. Each value materializes to a separate annotation token, while the resolver retains effective annotations by name. The last notify token replaces earlier ones.

Current bridge limitation

Use one Annotation::Notify value per sanction. The current ALAM constructor cannot express the multi-positional command form @notify(staff, audit).

Dry-run behavior

A dry-run sanction resolves the channel but does not dispatch a sanction notification. Notification delivery occurs only after live persistence completes.

Implementation notes, non-normative

The ALAM constructor is declared as:

new Ctor(
    "Notify",
    List.of(new Field("channel", TEXT))
)

Runtime materialization creates one positional token:

case "Notify" ->
    out.add(directive(
        "notify",
        ((Value.TextV) a.field(0)).value()
    ));

The resolver normalizes the effective channel list:

final String[] out =
    new String[a.positional().size()];

for (int i = 0; i < out.length; i++) {
    out[i] = a.positional()
        .get(i)
        .toLowerCase(Locale.ROOT);
}
  • STAFF and CONSOLE delivery is marshalled to the Paper main thread.
  • WEBHOOK delivery runs through the virtual-thread asynchronous scheduler.
  • Webhook HTTP requests have connection and request deadlines.
  • Delivery failure is logged and never rolls back an already persisted sanction.
