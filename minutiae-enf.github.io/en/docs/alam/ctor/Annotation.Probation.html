Annotation::Probation

constructor

Opens a finite window on the persisted sanction record during which later precedent under the same rule or joinder chain is treated as aggravated.

ContainerAnnotation
CategoryPrecedent lifecycle constructor

Syntax

  • (1)Positional construction.
  • (2)Named-field construction.

Fields

FieldTypeRequiredDescription
window Duration Yes Interval from the effective issue time during which later precedent is marked as being in probation.

Constraints

  • The constructor requires exactly one primitive Duration.
  • The system actor must hold minutiae.annotation.probation.
  • A zero window is type-correct but is not active after ordinary current-time comparison.
  • The constructor does not require an escalation ladder, but without escalation policy its aggravating effect may not change the selected duration.
  • The window is stored on the sanction record that carries the annotation. It is not a global account flag.
  • Probation applied during sanction issuance is distinct from probation attached to a pardon or time-served lift.

Semantics

When a live sanction is persisted, the executor computes:

probation_until =
    issued_at + window

issued_at is the effective issue timestamp after any backdate offset. A backdated sanction therefore opens a probation window relative to the backdated legal timeline, not relative to database insertion time.

A future precedent query reports active probation when at least one eligible record satisfies:

probation_until > now

The ordinary subject-and-rule query also requires that the record:

  • belongs to the same subject;
  • cites the same primary rule;
  • has not been vacated;
  • has not decayed out of precedent.

Aggravation policies

When precedent reports in_probation = true, runtime configuration selects one of two policies.

ModeCalculationEffect
Step index = min(priorSanctions + 1, ladder.length - 1) Advances one additional escalation rung.
Multiplier duration = round(duration × configuredMultiplier) Scales the selected finite duration.

A permanent selected duration is not multiplied. It remains permanent.

Current sanction distinction

The executor loads precedent before it persists the new sanction. Consequently, a probation window opened by the current sanction cannot aggravate that same sanction. It affects later precedent queries only.

Postcondition

After successful persistence, the sanction record carries one absolute probation deadline. No background task is scheduled. Later queries determine whether the deadline is still active.

Warning downgrade

The executor calculates probation_until independently of the warn-first downgrade. If the current offense is persisted as Measure::WARN, the warning row may still carry the probation deadline and may aggravate a later offense while that deadline remains active.

Interaction with decay

The ordinary precedent query applies decay filtering to the same rows used to detect probation. If the record decays before its probation deadline, it no longer contributes the active probation flag to that query.

Joinder-chain precedent

When a sanction is woven under a parent case, the executor may query the whole joinder chain. In that path, any eligible chain member with an unexpired probation deadline can mark the chain precedent as being in probation.

Implementation notes, non-normative

Materialization uses the ordinary command annotation bridge:

case "Probation" ->
    out.add(directive("probation", dur(a)));

The executor calculates the absolute deadline:

final long probationUntil =
    r.probationFor() != null
        ? issuedAt + r.probationFor().millis()
        : 0L;

Ordinary precedent derives the active flag with an aggregate:

MAX(
    CASE
        WHEN probation_until > ?
        THEN 1
        ELSE 0
    END
) AS probation

The result is projected to one Java value:

new Precedent(
    rs.getInt(1),
    rs.getInt(2),
    rs.getInt(3) != 0
)
  • The query returns only one boolean probation state, not the record that supplied it.
  • Step mode and multiplier mode are configured by EscalationConfig, not by the ALAM constructor.
  • Multiplier arithmetic uses Math.round(double) and does not use exact integer multiplication.
  • The sanction decision trace stores the inProbation flag that was consulted for the current decision.
  • No maintenance service clears expired probation deadlines; comparison with now makes them inactive lazily.

Example

.escalation = Escalation::Steps({
    Duration::Fixed(1d),
    Duration::Fixed(7d),
    Duration::Fixed(30d),
}),
.annotations = {
    Annotation::Escalate,
    Annotation::Probation(14d),
},

The current sanction opens a 14-day probation window. A later eligible offense inside the window receives the configured extra-step or multiplier aggravation.
