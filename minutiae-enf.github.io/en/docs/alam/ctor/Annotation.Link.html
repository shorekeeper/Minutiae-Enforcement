Annotation::Link

constructor

Attaches one opaque external reference string to the persisted sanction record.

ContainerAnnotation
CategoryCommand-scoped descriptor constructor
PayloadOne Text reference
Layout scopeRejected
Command form@link(reference)
Persisted fieldbans.link
Runtime stateNone

Syntax

Annotation::Link(reference)                                (1)

Annotation::Link {
    .ref = reference,
}                                                           (2)
  • (1)Positional construction.
  • (2)Named-field construction. The field name is ref.

Fields

FieldTypeRequiredDescription
ref Text Yes Opaque external reference such as a ticket identifier, case-system key, or URL.

Constraints

  • The constructor requires exactly one Text value.
  • The constructor is command-scoped and produces R005 in a stamped Layout.
  • The system actor must hold minutiae.annotation.link.
  • The compiler and resolver do not validate URI syntax, scheme, host, ticket format, or reachability.
  • An empty reference is type-correct and accepted by the current positional schema.
  • One resolved sanction stores at most one link string.

Semantics

The constructor materializes to:

@link(reference)

The shared resolver validates the annotation shape, checks permission, and copies the first positional value into ResolvedSanction.link. The executor persists that value in the sanction row.

Opaque-reference rule

ALAM assigns no semantics to the reference text. It does not fetch the target, verify evidence, authorize access, or infer a protocol from the content.

Display and projections

The database schema retains the link as nullable text. Not every current read projection exposes the field. In particular, the compact SanctionView projection used by several command and web-panel paths does not currently include bans.link.

Persisting a link therefore guarantees durable storage but does not guarantee that every inspection surface displays it in the current release.

Duplicate values

If several link annotations reach the resolver, effective annotations are merged by annotation name. The final effective link token supplies the persisted value.

Dry-run behavior

A dry-run resolves and validates the reference but persists no sanction row, so no link is stored.

Implementation notes, non-normative

The constructor field is named ref in the ALAM sum:

new Ctor(
    "Link",
    List.of(new Field("ref", TEXT))
)

Materialization creates one annotation token:

case "Link" ->
    out.add(directive("link", text(a, 0)));

The resolver extracts the effective value:

final String link =
    firstPositional(effective, "link");

Persistence writes it directly:

final SanctionRow row = new SanctionRow(
    uuid.toString(),
    r.layoutKey(),
    r.rule(),
    r.reason(),
    issuedAt,
    expiresAt,
    staff,
    renderAnnotations(r),
    active,
    measure.name(),
    stayed ? 1 : 0,
    stayUntil,
    r.link(),
    behaviourMask,
    // Remaining persistence fields.
);

The corresponding SQL parameter is nullable:

ps.setString(13, row.link());
Complexity
Materialization
O(L), where L is the reference length.
Validation
O(1) structural checks; no URI parsing or network access.
Storage
O(L) nullable database text.

Example

.annotations = {
    Annotation::Reason("Evidence reviewed by staff"),
    Annotation::Link(
        "https://cases.example.invalid/tickets/ME-412"
    ),
},
