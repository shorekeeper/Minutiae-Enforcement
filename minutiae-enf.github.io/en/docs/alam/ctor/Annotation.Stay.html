Annotation::Stay

constructor

Records a sanction in a deferred state and makes it eligible for activation by a later sanction during a finite stay window.

ContainerAnnotation
CategoryProcedural lifecycle constructor
PayloadOne finite Duration
Layout scopeRejected
Command form@stay(window)

Syntax

Annotation::Stay(window)                                   (1)

Annotation::Stay {
    .window = window,
}                                                           (2)
  • (1)Positional construction.
  • (2)Named-field construction.

Fields

FieldTypeRequiredDescription
window Duration Yes Period during which a subsequent sanction may activate the deferred sanction.

Constraints

  • The constructor requires exactly one primitive Duration.
  • The constructor is command-scoped and produces R005 when retained in a stamped Layout.
  • The system actor must hold minutiae.annotation.stay.
  • A stayed sanction cannot also be recorded as a suspended sentence.
  • A zero window is type-correct but leaves no practical interval in which activation can succeed.

Semantics

A live sanction carrying the constructor is persisted with:

active = 0
stayed = 1
stay_until = execution_time + window

The sanction record exists immediately, contributes to audit and docket history, and may contribute to precedent according to the ordinary precedent query. Its enforcement mechanism is not applied while it remains stayed.

Activation

After a new sanction is persisted, the executor activates every stayed sanction for the same subject whose stay window remains open:

subject matches
&& stayed == true
&& stay_until > now

Activation is subject-wide, not rule-scoped. A later sanction under a different rule may activate the stay.

Activation changes:

active = 1
stayed = 0

The stored measure, expiry, behavior mask, and reason remain unchanged.

Expiry timeline

The sanction's enforcement expiry is calculated from its issue timestamp when it is first recorded. Activating a stay does not restart or extend that expiry.

A long stay on a short temporal sanction can therefore activate a sanction whose enforcement expiry has already passed. Storage activation itself does not add an expiry predicate.

Effect application

The immediate issuance path skips applyMechanism for a stayed outcome. When a later sanction calls activateStays, the current executor records the activation count for reporting. The shown implementation does not independently replay each activated sanction's player effect in the same activation query.

Operational detail

A stay is a persisted procedural state, not a paused duration. Design stay windows and sanction durations together.

Implementation notes, non-normative

Materialization produces the inline annotation:

case "Stay" ->
    out.add(directive("stay", dur(a)));

Issuance computes the state:

final boolean stayed =
    r.stayFor() != null;

final long stayUntil = stayed
    ? now + r.stayFor().millis()
    : 0L;

final int active =
    activeFlag(measure, stayed, suspended);

Storage activation is one indexed update:

UPDATE bans
SET active = 1,
    stayed = 0
WHERE uuid = ?
  AND stayed = 1
  AND stay_until > ?
  • Activation occurs after the new sanction and its auxiliary records are persisted.
  • The activation count is included in the new sanction's execution outcome.
  • Expired stay windows are not activated by this query.
  • A stayed sanction is excluded from active connection-ban and behavioral-state queries until activation.
Complexity
Issuance

Activation

Example

    Annotation::Stay(7d),
    Annotation::Reason("Activated on further misconduct"),
