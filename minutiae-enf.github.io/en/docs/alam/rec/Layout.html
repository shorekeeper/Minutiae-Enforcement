Layout

record

A total compile-time descriptor that may be stamped into the ordinary layout registry and later selected by manual or automatic sanction resolution.

Fields

FieldTypeRequiredDescription
key Text Yes Invocation key under which the stamped layout enters the registry.
rule Rule Yes Primary external rule identifier used by sanctions selecting the layout.
reason Text Yes Default sanction reason supplied by the layout.
measure Measure Yes Default enforcement mechanism.
duration Duration term Yes Fixed finite duration or the non-fixed term required by the measure's temporal class.
escalation Escalation Yes Absent or explicit ordered duration ladder.
annotations Annotation[] Yes Default configuration-scoped annotations applied when the layout is selected.

Constraints

Structural requirements
  1. Every field is supplied exactly once.
  2. No unknown field is supplied.
  3. Every field expression has the exact declared type.
  4. Construction and all producing transform logic remain pure.
Post-evaluation verification
  1. rule exists in the external rule registry.
  2. measure and duration satisfy the temporal invariant.
  3. An explicit escalation ladder is non-empty, contains positive fixed rungs, and places a permanent rung only at the end.
  4. key is not blank.
  5. Every annotation constructor is valid in configuration scope.

Semantics

Constructing a Layout value does not by itself publish a registry entry. A layout is stamped only when an expand declaration produces it.

Publication rule

A const Layout remains an ordinary compile-time value. It is not included in Evaluator.Output.layouts and is not visible to the runtime layout registry unless an expansion produces that value.

key

The key is used without the command prefix. A stamped key "chat-flood" is selected on the command surface as ::chat-flood.

Keys must be unique within one compilation unit. Across ALAM files, the first accepted producer in lexical file-name order wins. A later collision is logged and dropped.

A key colliding with a file-authored entry in layouts.yml is also dropped. The file-authored definition remains authoritative.

rule

The rule identifier becomes the layout's primary rule. It supplies precedent scope, rule description fallback in ordinary file-authored layouts, and the principal citation persisted with sanctions.

reason

The reason is mandatory in ALAM. An empty string remains a present string after lowering.

Empty reason behavior

ALAM lowering does not convert empty text to null. Because the resolver treats any non-null layout reason as present, an empty ALAM layout reason does not fall back to the external rule description.

measure and duration

The verifier enforces the temporal relationship before lowering:

Measure classRequired termLowered duration
TemporalDuration::Fixed with positive spanFinite DurationSpec
InstantaneousDuration::Permanentnull
PermanentDuration::Permanentnull; resolver derives permanent duration from measure

escalation

Escalation::Steps lowers to an ordered DurationSpec[]. Escalation::None lowers to an absent ladder in the raw stamped definition and ultimately to an empty array in the resolved layout.

The complete array is retained by the layout registry and is therefore available to sanction execution. This is the supported path for transporting an ALAM-defined escalation ladder.

annotations

Each constructor is converted to a configuration-surface RawAnnotation. The verifier rejects command-scoped constructors before lowering.

Layout annotations become defaults. Inline command annotations may replace or negate them when ordinary command semantics permit it. Permissions are still checked against the sender selecting the layout.

No ALAM inheritance fields

The built-in ALAM record has no extends or private-key field. Lowering always creates:

isPrivate = false
extendsKey = null

An underscore-prefixed ALAM key is therefore not automatically private. Private inheritance bases remain a feature of file-authored layouts.yml definitions.

Registry ingestion

After all accepted ALAM units are planned, their stamped definitions are supplied to LayoutRegistry. The registry merges them with file-authored definitions, resolves structural inheritance, validates referential completeness, and publishes an immutable invocable map.

Whole-unit publication

If the source unit has an error in any compilation phase, no layout produced by that unit is stamped. Accepted sibling source files remain available.

Effect

pure Layout construction, transforms, matrix rows, and expansion are compile-time pure. Selecting the resulting layout later may cause queries and enforcement effects through the ordinary sanction path.

Implementation notes, non-normative

The built-in record is declared as:

public static final Rec LAYOUT =
    new Rec("Layout", List.of(
        new Field("key", TEXT),
        new Field("rule", RULE),
        new Field("reason", TEXT),
        new Field("measure", MEASURE),
        new Field("duration", DURATION_TERM),
        new Field("escalation", ESCALATION_RESOLVED),
        new Field("annotations", new ListT(ANNOTATION))
    ));

expand recognizes the produced record by nominal type name and enforces unit-local key uniqueness:

case Value.RecordV rec
        when rec.type().name()
            .equals(Builtins.LAYOUT.name()) -> {
    final String key =
        ((Value.TextV) rec.field("key")).value();

    if (!layoutKeys.add(key)) {
        error(
            "V016",
            x.pos(),
            "duplicate layout key '" + key
                + "' produced by expansion"
        );
        continue;
    }

    layouts.add(rec);
}

Lowering constructs a registry definition with no inheritance metadata:

return new LayoutDefinition(
    key,
    false,
    null,
    rule,
    reason,
    measure,
    duration,
    ladder,
    List.copyOf(annotations)
);

Cross-file ALAM collisions use first-wins publication:

if (!layoutKeys.add(def.key())) {
    log.warn(
        "alam",
        "%s: stamped layout '%s' dropped: "
            + "key already stamped by an earlier unit",
        file.getName(),
        def.key()
    );
    continue;
}

File-authored definitions remain authoritative when the layout registry ingests stamped entries:

if (definitions.putIfAbsent(
        def.key(),
        def) != null) {
    log.warn(
        "layouts",
        "stamped layout '%s' ignored: "
            + "key already defined in '%s'",
        def.key(),
        fileName
    );
}
  • Accepted unit plans and stamped definitions are published as immutable lists.
  • The layout registry publishes its invocable map through one volatile reference.
  • A stamped layout is treated like an ordinary raw definition after ingestion.
  • The registry remains responsible for final rule existence and temporal completeness checks, even though ALAM verification has already established them for stamped data.

Example

schema LayoutRow {
    Text key;
    Rule rule;
    Text reason;
    Measure measure;
    Duration span;
}

matrix<LayoutRow> layouts {
    row(
        "chat-flood",
        P.2.1,
        "Repeated message flooding",
        Measure::MUTE,
        3h
    );
}

transform<LayoutRow -> Layout> stamp_layout =
    (entry) -> Layout {
        .key = entry.key,
        .rule = entry.rule,
        .reason = entry.reason,
        .measure = entry.measure,
        .duration = Duration::Fixed(entry.span),
        .escalation = Escalation::Steps({
            Duration::Fixed(entry.span),
            Duration::Fixed(entry.span * 4),
            Duration::Permanent,
        }),
        .annotations = {
            Annotation::Escalate,
            Annotation::Notify("staff"),
        },
    };

expand stamp_layout(layouts);
