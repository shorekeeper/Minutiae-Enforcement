Constraints

Layout scope

A stamped Layout may contain only configuration-scoped constructors:

Notify, Evidence, Escalate, Silent, Shadow, Ghost,
Rubberband, Transcript, WarnFirst, Decay, Tariff, Probation

Annotation::Reason, Annotation::Link, and Annotation::Stay are command-scoped and produce R005 when retained in a stamped layout.

  • Every constructor payload must have its exact declared type.
  • Constructors with data require exactly one argument.
  • Nullary constructors accept no arguments.
  • The containing annotations field is a homogeneous Annotation[] list.
  • ALAM establishes constructor shape and type correctness. Command-level parameter rules and permission checks are applied after materialization.
  • Annotation::Tariff is meaningful only with a temporal measure.
  • Annotation::Stay cannot coherently combine with a suspended-sentence directive introduced outside this ALAM sum.

Semantics

An ALAM annotation is a typed constructor value. It is not the same Java object as an annotation token used by commands and layouts. Before sanction resolution, lowering or materialization translates each constructor into a RawAnnotation or scalar override.

Annotation::Evidence

Annotation::Evidence

Produces the bare @evidence token. In the command annotation schema, the bare form is valid and represents the default evidence requirement.

Annotation::Escalate

Annotation::Escalate

Engages duration escalation. When supplied through a layout, the sanction executor may select a rung from that layout's escalation array. An explicit duration override bypasses rung selection.

Annotation::Silent

Annotation::Silent

Suppresses the public sanction broadcast. It does not suppress direct reports to the issuer, persistence, audit, or explicitly requested notification channels.

Annotation::Shadow

Annotation::Shadow

Requests covert isolation. The resolved command relationship matrix normally expands shadow to silent. The runtime behavioral mask receives the SHADOWED bit.

A shadowed connection-blocking sanction permits login and applies isolation after join instead of showing the sanction refusal screen.

Annotation::Ghost

Annotation::Ghost

Applies the GHOSTED behavioral bit. The player is hidden from other viewers and their world interactions are cancelled. Unlike shadow, chat is not redirected to the sender-only viewer set by this flag alone.

Annotation::Rubberband

Annotation::Rubberband

Applies the RUBBERBAND behavioral bit. A movement crossing a block boundary is reverted when its three-dimensional displacement exceeds the configured leash.

Annotation::Transcript

Annotation::Transcript

Requests a snapshot of the subject's bounded in-memory chat ring. The snapshot is taken before sanction persistence completes and is stored against the generated sanction identifier.

Materialization rules

Most constructors become ordinary command annotations. Two constructors receive special treatment:

ConstructorMaterialized resultReason
Annotation::Reason overrides["reason"] The manual command surface treats reason as a scalar override.
Annotation::Escalate @escalate Engages the resolver's ordinary escalation path.
Duplicate constructor behavior

The ALAM list type permits repeated annotation constructors. The downstream resolver stores effective annotations by command name, so later materialized tokens of the same name replace earlier tokens. Multiple Annotation::Notify values therefore do not accumulate channels through the current command bridge; the final effective @notify token wins.

Implementation notes, non-normative

The built-in sum is a closed constructor list:

public static final Sum ANNOTATION = new Sum(
    "Annotation",
    List.of(
        new Ctor("Notify",
            List.of(new Field("channel", TEXT))),
        new Ctor("Evidence", List.of()),
        new Ctor("Escalate", List.of()),
        new Ctor("Silent", List.of()),
        new Ctor("Shadow", List.of()),
        new Ctor("Ghost", List.of()),
        new Ctor("Rubberband", List.of()),
        new Ctor("Transcript", List.of()),
        new Ctor("Reason",
            List.of(new Field("text", TEXT))),
        new Ctor("Link",
            List.of(new Field("ref", TEXT))),
        new Ctor("WarnFirst",
            List.of(new Field("count", INT))),
        new Ctor("Decay",
            List.of(new Field("window", DURATION))),
        new Ctor("Stay",
            List.of(new Field("window", DURATION))),
        new Ctor("Tariff",
            List.of(new Field("floor", DURATION))),
        new Ctor("Probation",
            List.of(new Field("window", DURATION)))
    )
);

Runtime sanction materialization uses an exhaustive constructor switch:

switch (a.ctor()) {
    case "Notify" ->
        out.add(directive("notify", text(a, 0)));
    case "Evidence" ->
        out.add(flag("evidence"));
    case "Reason" ->
        overrides.put("reason", text(a, 0));
    case "WarnFirst" ->
        out.add(directive("warn-first",
            Long.toString(((Value.IntV) a.field(0)).value())));
    case "Decay" ->
        out.add(directive("decay", dur(a)));
    // Remaining constructors follow the same bridge.
}
  • Materialized annotations pass through SanctionResolver under the system actor's permission predicate.
  • The resolver applies configured implication, requirement, and conflict relationships to the resulting annotation mask.
  • Behavioral bits are derived later by SanctionExecutor.behaviourMaskOf.
  • Layout lowering and runtime sanction materialization use separate methods because layout scope is narrower.
  • The default branch of runtime materialization ignores an unknown constructor. Under the closed, verified built-in sum, that branch is unreachable unless compiler and runtime versions become inconsistent.
Complexity
Construction
O(1) per constructor value.
List materialization
O(N) for N annotation values.
Relationship expansion
O(B), where B is the number of set bits in the 64-bit annotation mask.
Storage
O(N) before effective-name merging; at most one effective token per annotation name afterward.

Example

template<Measure M, Duration D>
verdict notified = (event) -> Sanction {
    .target = event.subject,
    .cite = { P.2.1 },
    .measure = M,
    .duration = Duration::Fixed(D),
    .escalation = Escalation::None,
    .annotations = {
        Annotation::Notify("staff"),
        Annotation::Evidence,
        Annotation::Transcript,
        Annotation::Reason("Repeated message flooding"),
    },
    .attribution = Attribution::System,
    .dry_run = DryRun::InheritSafety,
};
