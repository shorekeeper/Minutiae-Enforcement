Escalation

sum

The closed descriptor type representing either the absence of a duration ladder or an explicit ordered ladder of duration terms.

Syntax

  • (2)Declares an ordered list of fixed or permanent duration terms.

Fields

Constraints

Valid layout ladder
  1. The steps list is non-empty.
  2. Every Duration::Fixed rung contains a positive finite span.
  3. Duration::Permanent, when present, is the final rung.
  4. No rung follows a permanent rung.

These residual ladder invariants are verified for stamped Layout values after compile-time evaluation. A violation produces R003.

The type system establishes only that the field is a homogeneous list of duration terms. It cannot establish non-emptiness, positivity, or final-rung placement before evaluation.

Semantics

Escalation::None

Escalation::None

Specifies no ladder. A layout lowered with this constructor exposes no escalation array to the ordinary sanction resolver.

The constructor does not prevent an annotation or command directive named escalate from being present. Such a directive has no useful ladder to select from and therefore cannot derive a rung from this descriptor.

Escalation::Steps

The ordered list represents increasingly severe duration outcomes. Operational execution selects a rung using prior non-warning sanctions:

index = min(prior_sanctions, steps.length - 1)

The final rung therefore repeats for every precedent count beyond the list length.

When the subject is inside an active probation window, policy may either advance one additional rung or multiply the selected finite span. That behavior belongs to runtime escalation configuration rather than the ALAM sum itself.

Descriptor transport distinction

A ladder stored in a stamped Layout is lowered to the layout registry and is available to sanction execution. A ladder placed directly in a runtime-produced Sanction is currently materialized as the presence of the escalate directive; its individual rung values are not transported through the manual command surface.

This distinction means that reusable duration ladders should normally be stamped into layouts. A verdict that directly constructs a sanction should not rely on transporting an arbitrary private ladder to the resolver.

Implementation notes, non-normative

The built-in type is declared with a homogeneous list field:

public static final Sum ESCALATION_RESOLVED =
    new Sum("Escalation", List.of(
        new Ctor("None", List.of()),
        new Ctor("Steps", List.of(
            new Field("steps", new ListT(DURATION_TERM))
        ))
    ));

Layout lowering converts the value to a Java array:

final DurationSpec[] ladder;
if (escalation.ctor().equals("Steps")) {
    final List<Value> steps =
            ((Value.ListV) escalation.field(0)).items();
    ladder = new DurationSpec[steps.size()];
    // Each Fixed or Permanent term is lowered in order.
} else {
    ladder = null;
}

At sanction execution, the selected rung is computed from persisted precedent:

int index = Math.min(
        precedent.priorSanctions(),
        r.escalation().length - 1
);
base = r.escalation()[index];
  • The resolver stores a layout ladder as DurationSpec[].
  • An explicit command duration= override bypasses ladder selection.
  • A tariff annotation may raise the selected finite duration to a configured floor.
  • A permanent rung is represented by DurationSpec.PERMANENT.
Complexity
Construction
O(N) time and storage for N duration terms.
Verification
O(N) over the ladder.
Rung selection
O(1) after lowering to an array.

Example

const Escalation chat_ladder = Escalation::Steps({
    Duration::Fixed(1h),
    Duration::Fixed(1d),
    Duration::Fixed(7d),
    Duration::Permanent,
});

const Layout repeated_flood = Layout {
    .key = "repeated-flood",
    .rule = P.2.1,
    .reason = "Repeated message flooding",
    .measure = Measure::MUTE,
    .duration = Duration::Fixed(1h),
    .escalation = chat_ladder,
    .annotations = {
        Annotation::Escalate,
    },
};
