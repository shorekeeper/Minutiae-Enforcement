Syntax

  • (2)Partitioned sliding recurrence window.

Constraints

Rule-shape correspondence
ConstructorRequired declared eventsAdditional requirement
Trigger::Atomic Exactly one None
Trigger::Repeated Exactly one Positive finite span and count in 1 through 10000
Trigger::Sequence Two or more One ordered step per event, with matching event types
  • The trigger expression is constructed under the pure effect allowance.
  • The complete expression must have nominal type Trigger.
  • A constructor with fields must receive every required field exactly once.
  • Window spans must be finite because the runtime stores epoch-millisecond boundaries.

Semantics

The trigger expression is evaluated while the source unit is compiled. A rule does not retain trigger syntax or re-evaluate constructor arguments for each event. The resulting variant is validated against the rule's event list and lowered to primitive runtime plan data.

ConstructorGuard timingStateFiring condition
Trigger::Atomic Rule guard on each candidate None The rule guard returns true
Trigger::Repeated Rule guard before window insertion Timestamp ring per partition The admitted occurrence count reaches the threshold inside the span
Trigger::Sequence Step guards during admission; rule guard after completion Partial index and captured events per partition All steps complete in order and the final rule guard returns true

Atomic dispatch

An atomic rule receives each event independently. No state is retained between events. The event is adapted to the rule's one event record and passed directly to the rule guard.

Repeated dispatch

A repeated rule evaluates its rule guard before recording the occurrence. Only guard-admitted events enter the recurrence ring. A successful recurrence clears the ring, so the next firing starts from zero.

Sequence dispatch

A sequence evaluates the guard attached to the next expected Step. A matching event advances the partial state and is captured. When the last step completes, the rule guard and verdict receive one record per declared event.

If a later step is expected but the incoming event satisfies the first step, the partial match re-anchors at the new event. This prevents a stale partial from blocking overlapping sequences.

Throttle interaction

Passing a trigger does not by itself execute the verdict. The owning automaton must also acquire one unit from its per-minute throttle. A throttle breach self-mutes the entire automaton.

Implementation notes, non-normative

The compile-time value is Value.VariantV`. Evaluation validates its shape before lowering:

switch (trigger.ctor()) {
    case "Atomic"   -> validateAtomic(events);
    case "Repeated" -> validateRepeated(trigger, events);
    case "Sequence" -> validateSequence(trigger, events);
}

Lowering produces the sealed plan hierarchy:

public sealed interface TriggerPlan
        permits TriggerPlan.Atomic,
                TriggerPlan.Repeated,
                TriggerPlan.Sequence {
}
  • TriggerPlan.Atomic is a singleton and carries no fields.
  • TriggerPlan.Repeated carries an int count, a long millisecond span, and a partition selector.
  • TriggerPlan.Sequence carries a millisecond span, partition selector, and array of lowered steps.
  • DispatchEngine uses a Java pattern-switch over the sealed hierarchy, so adding another trigger form requires a deliberate compiler and runtime change.
  • The engine indexes a sequence rule under every step event kind, while atomic and repeated rules are indexed under their single event kind.
Complexity summary
Atomic
O(G) time and O(1) trigger storage, where G is rule-guard cost.
Repeated
O(G + count) time per admitted event and O(count) storage per active partition.
Sequence
O(SG + G + V) over a completed match, with O(S) captured events per active partition, where S is step count.

Example

rule<Login, Chat> immediate_appeal {
    condition {
        trigger = Trigger::Sequence {
            .within = 10m,
            .group_by = GroupBy::Subject,
            .steps = {
                Step<Login> {
                    .guard = Guard::Always,
                },
                Step<Chat> {
                    .guard = (event) ->
                        event.message ~ "\\bappeal\\b";
                },
            },
        };
        guard = (login, chat) ->
            login.subject == chat.subject;
    }
    => verdict = appeal_verdict<P.1.1>;
}
