The closed descriptor type selecting either configured system attribution or an explicit staff display name for an automatically produced sanction.

Syntax

Attribution::System                                      (1)

Attribution::Staff(name)                                 (2)
  • (1)Requests attribution to the configured automation actor.
  • (2)Requests attribution to an explicit staff display name.
attribution =
      Attribution::System
    | Attribution::Staff(staff-name) ;

Fields

ConstructorFieldTypeDescription
Attribution::System - - Uses the configured ALAM system actor name when the actor is allowed to apply attribution.
Attribution::Staff name Text Supplies an explicit staff attribution name through the command annotation surface.

Constraints

  • The attribution field is mandatory in every Sanction construction.
  • Attribution::System is nullary.
  • Attribution::Staff requires exactly one Text value.
  • Neither constructor proves that the supplied name identifies a real online or historical player.
  • Applying an explicit attribution is permission-gated by the downstream minutiae.annotation.as grant.
  • Attribution changes the recorded issuer name. It does not expand the system actor's authority.

Semantics

Attribution::System

Attribution::System

Requests attribution to the configured ALAM system actor, normally SYSTEM. The descriptor does not directly write that name into the sanction command.

During firing, the dispatch engine checks whether the system actor has the minutiae.annotation.as permission. If granted, it appends an @as(configured-system-name) token before sanction resolution.

If the actor lacks that permission, no attribution directive is added. The sanction executor then falls back to the name of the actual command sender used as the dispatch sink, normally the server console sender.

Permission boundary

Attribution::System does not bypass the permission predicate. The configured system name is applied only if automation is explicitly granted the same attribution permission required by a manual command.

Attribution::Staff

Materializes directly to @as(name). The shared sanction resolver validates the annotation and requires the system actor to hold minutiae.annotation.as.

The supplied text is treated as an attribution label. The ALAM compiler and sanction resolver do not perform an account or staff-directory lookup.

Recorded effects

The effective attribution name is stored in the sanction row's staff column. It is subsequently used for:

  • operator-facing reports and audit text;
  • public sanction broadcasts;
  • notification summaries;
  • foreign-lift ownership checks;
  • staff-based docket and bulk-lift criteria;
  • rank display resolution for the issuing sender.

The rank shown in a broadcast is resolved from the actual issuing sender, while the staff name may be overridden. These are separate values.

Implementation notes, non-normative

SanctionMaterializer distinguishes the constructors:

boolean systemAttribution = false;

final Value.VariantV attribution =
        (Value.VariantV) sanction.field("attribution");

if (attribution.ctor().equals("Staff")) {
    annotations.add(directive(
        "as",
        ((Value.TextV) attribution.field(0)).value()
    ));
} else {
    systemAttribution = true;
}

DispatchEngine.withPosture conditionally applies the configured system name:

if (m.systemAttribution()
        && safety.systemActor().name() != null
        && safety.systemActor()
                .has("minutiae.annotation.as")) {
    annotations.add(new RawAnnotation(
        false,
        "as",
        List.of(safety.systemActor().name()),
        Map.of()
    ));
}

The sanction executor chooses the final recorded name:

final String staff =
    r.attributedStaff() != null
        ? r.attributedStaff()
        : issuer.getName();
  • The system actor's permission predicate is still used for complete sanction resolution.
  • The issuer object remains the sink for reports and error messages even when the stored attribution name differs.
  • An explicit attribution affects later foreign-lift checks because those compare the stored staff name with the lifting sender name.

Example

template<Measure M>
verdict automatic = (event) -> Sanction {
    .target = event.subject,
    .cite = { P.5.1 },
    .measure = M,
    .duration = Duration::Permanent,
    .escalation = Escalation::None,
    .annotations = {},
    .attribution = Attribution::System,
    .dry_run = DryRun::InheritSafety,
};
