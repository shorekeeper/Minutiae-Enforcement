Event type

type

The nominal family of event record types and the distinguished internal base event used for event-polymorphic function positions.

Syntax

event-type =
      Chat
    | Break
    | Login
    | Evasion ;

rule-event-list =
    event-type { "," event-type } ;

step-event-argument =
    "Step" "<" event-type ">" ;

Fields

Every concrete event contains the common base fields:

FieldTypeDescription
subject Account Stable account identity of the event subject.
subject_name Text Display name captured with the event facts.

Concrete events append their own fields after these common fields.

Constraints

  • A rule event list and Step<...> argument must name a known concrete event.
  • The distinguished base Event is used by the type system for event-polymorphic templates. It is not currently accepted as a concrete event in a rule header.
  • Event types are nominal. Two concrete events are not assignable to each other.
  • Every concrete event is assignable to the distinguished base event.
  • Only fields declared by the statically known event type may be accessed.

Semantics

An event type defines the record shape presented to a rule guard, sequence step guard, and verdict. The shape is selected during lowering and does not depend on the runtime contents of the fact map.

The base event exists to type bodies that may operate on any event. Such a body can access subject and subject_name only. It cannot access a field such as message because that field is not present on every event.

Event subtyping is the only nominal aggregate subtyping axis in ALAM. A concrete event can flow to an Event parameter; the reverse assignment is not valid.

Under the Java implementation

Each event is a Type.Event containing its name and ordered Type.Field list. Builtins.ANY_EVENT is the distinguished base shape.

Lowering interns one Type.Rec shape per event name. EventAdapter walks that shape field by field and creates a Value.RecordV. Missing numeric, boolean, or text facts become neutral values.

The runtime event object is EventFacts and contains no Bukkit or Paper type. Live listeners and the simulation command therefore feed the same interpreter representation.

Example

template<Measure M>
verdict for_subject = (event) -> Sanction {
    .target = event.subject,
    .cite = { P.1.1 },
    .measure = M,
    .duration = Duration::Permanent,
    .escalation = Escalation::None,
    .annotations = {},
    .attribution = Attribution::System,
    .dry_run = DryRun::InheritSafety,
};

The template body is checked against the base event type and therefore uses only subject.

Diagnostics

E003An event value is supplied to an incompatible concrete event position.
E015A body accesses a field absent from the statically known event shape.
E025A rule, step, or structural rule shape names an unknown event.
V002Sequence step events do not correspond to the containing rule event list.
