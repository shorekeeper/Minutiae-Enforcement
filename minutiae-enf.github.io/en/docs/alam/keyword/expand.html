MEMinutiae Enforcement docs

expand

keyword

Applies one pure unary transform to every value of an ordered compile-time list and contributes each result as a generated layout or rule artifact.

CategoryTop-level artifact-producing declaration
Introduces a source nameNo
InputOne homogeneous list, normally a matrix
OperationPure unary transform
Accepted outputsLayout or structural Rule<Events...>
Evaluation timeCompile time
OrderingInput order
Partial publicationNever; any error rejects the complete source unit

Syntax

expand transform-expression(input-list);
expand-declaration =
    "expand" application-expression ";" ;

application-expression =
    transform-expression "(" input-list ")" ;

Parameters

PartRequired typeDescription
Transform expression (T) -> Layout or (T) -> Rule<Events...> Pure unary function applied once to every input item.
Input expression T[] Ordered homogeneous list whose element type exactly matches the transform parameter.
Produced item Layout or structural Rule<Events...> Residual artifact appended to the evaluator output.

Constraints

Ill-formed if
  1. The expression following expand is not syntactically a call expression.
  2. The call supplies other than exactly one argument.
  3. The callee does not have a pure unary function type.
  4. The input is not a list of the callee's parameter type.
  5. The transform result is neither Layout nor structural Rule<Events...>.
  6. The application requires a query-capable operation.
  7. Any transform application fails during compile-time evaluation.
  8. Two generated layouts in the same unit have the same key.
  9. A generated artifact fails post-evaluation verification.
  • The input list may be empty.
  • The callee may be a named transform or a compatible composed pure function.
  • The input expression is evaluated once before element-wise application begins.
  • The transform value is evaluated once before iteration begins.
  • expand contributes artifacts but binds no value and cannot be referenced by later declarations.
  • Output artifact types are homogeneous because one transform has one result type.

Semantics

Evaluation order

For an input list containing values x0 through xN-1:

  1. Evaluate the transform expression.
  2. Evaluate the one input expression.
  3. For each index i from zero through N - 1, apply the transform to xi.
  4. Inspect the produced value.
  5. Append a layout or generated rule artifact to the corresponding residual collection.
  6. Continue with the next input item.
output[0] = transform(input[0])
output[1] = transform(input[1])
...
output[N - 1] = transform(input[N - 1])

The transform is not applied lazily. All results are produced before verification and lowering begin.

Input list

A matrix<Schema> is exposed as Schema[] and is the usual expansion input. However, expand is typed against a list rather than against a special matrix type. Any pure expression producing the exact compatible list type is admissible.

The current standard library provides no general mapping or filtering functions. A named matrix is therefore the practical source in most definitions.

Generated layouts

A produced Layout is appended to the unit's stamped-layout collection. Its key field is checked immediately against all layout keys previously produced by expansions in the same unit.

Unit-local layout-key ownership

The first generated layout with a key owns that key within the unit. A later duplicate produces V016. Although evaluation may continue after reporting the duplicate, the error prevents publication of the complete unit.

After unit verification, layout lowering creates LayoutDefinition values. Service publication then performs two additional ownership checks:

  1. Across accepted ALAM files, the first producer in lexical file-name order wins.
  2. A file-authored definition in layouts.yml wins over a stamped ALAM definition with the same key.

Cross-file collisions are service warnings rather than ALAM compiler diagnostics because each source unit is individually valid.

Generated rules

A produced structural rule is appended to the evaluator's generated-rule list. The evaluator synthesizes a display name from the expansion callee's base name and the zero-based input index:

base-name "#" index

make_rule#0
make_rule#1
make_rule#2

If the callee is a member-composition expression, the evaluator walks through member receivers to recover the leftmost base NameRef. When no simple base name is recoverable, it uses expand.

Current generated-rule publication limit

Generated rules are verified but are not addressable by automaton use or override. They are not inserted into the explicit rule symbol map, and lowering builds runtime automata only from resolved automaton membership. Under the current implementation, an expanded rule does not become an armed runtime rule.

The limitation has several concrete consequences:

  • a generated rule cannot be named in source after expansion;
  • its synthesized #index name is not a source identifier;
  • W003 does not report it as unused because that warning iterates explicit rules only;
  • its guard and verdict are still checked for external rule references by the verifier;
  • it does not appear in UnitPlan.automata unless future compiler wiring explicitly adds generated membership.

Empty input

An empty input list produces no artifacts and no diagnostic. The expansion declaration still evaluates successfully.

Failure during iteration

A transform application may fail after earlier elements have already produced temporary residual artifacts. For example, row zero may produce a layout before row one divides by zero.

Whole-unit guarantee

Temporary evaluator progress is never equivalent to publication. Any error-severity diagnostic prevents Planner from producing a UnitPlan. No rule or layout from that source file is armed or stamped.

Post-evaluation verification

Every generated layout is checked for:

  • external rule existence;
  • measure-duration agreement;
  • valid escalation ladder;
  • non-blank key;
  • configuration-scoped annotations only.

Every explicit and generated rule is walked for reachable external rule identifier literals in guards, verdicts, and sequence step guards.

Effect

pure The transform, input expression, and every element application are evaluated under the pure allowance.

Expansion cannot consult live precedent, online state, fingerprint score, or wall-clock time.

Implementation notes, non-normative

The parser requires a call-shaped expression:

final Expr application = expr();

if (!(application instanceof Expr.Call)) {
    report(
        "P007",
        "'expand' requires a transform application "
            + "of the form name(argument)"
    );
}

The parser reports P007 but still constructs the AST node. Because the diagnostic is an error, planning cannot proceed to successful publication.

Elaboration enforces one pure parameter and an accepted output:

if (!(expand.application()
        instanceof Expr.Call call)
        || call.args().size() != 1) {
    error(
        "E023",
        expand.pos(),
        "'expand' requires a single-argument "
            + "transform application"
    );

    throw Abort.INSTANCE;
}

final Type calleeType =
    infer(
        call.callee(),
        emptyScope,
        Effect.PURE
    );

if (!(calleeType instanceof Func function)
        || function.params().size() != 1
        || function.effect() != Effect.PURE) {
    error(
        "E023",
        call.callee().pos(),
        "expansion target must be a pure "
            + "single-parameter transform, not "
            + show(calleeType)
    );

    throw Abort.INSTANCE;
}

The accepted result test is nominal for Layout and structural-family based for rules:

final Type produced =
    function.result();

final boolean layout =
    same(produced, Builtins.LAYOUT);

final boolean rule =
    produced instanceof RuleT;

if (!layout && !rule) {
    error(
        "E023",
        call.callee().pos(),
        "expansion must produce Layout or Rule "
            + "values, not "
            + show(produced)
    );

    throw Abort.INSTANCE;
}

Evaluation applies the function element by element:

final Expr.Call call =
    (Expr.Call) expand.application();

final Value function =
    eval(call.callee(), global);

final Value.ListV input =
    (Value.ListV) eval(
        call.args().get(0),
        global
    );

final String base =
    expansionBaseName(call.callee());

int index = 0;

for (final Value row : input.items()) {
    final Value produced =
        apply(
            function,
            List.of(row),
            expand.pos()
        );

    switch (produced) {
        case Value.RecordV record
                when record.type()
                    .name()
                    .equals(Builtins.LAYOUT.name()) -> {
            final String key =
                ((Value.TextV) record.field("key"))
                    .value();

            if (!layoutKeys.add(key)) {
                error(
                    "V016",
                    expand.pos(),
                    "duplicate layout key '"
                        + key
                        + "' produced by expansion"
                );

                continue;
            }

            layouts.add(record);
        }

        case Value.RuleV rule ->
            expandedRules.add(
                rule.named(base + "#" + index)
            );

        default -> {
            error(
                "V013",
                expand.pos(),
                "expansion produced an unexpected "
                    + "value kind"
            );

            throw Abort.INSTANCE;
        }
    }

    index++;
}

The planner rejects temporary partial output immediately after evaluation diagnostics are checked:

final Evaluator.Output output =
    Evaluator.run(elaboration, diagnostics);

if (diagnostics.hasErrors()) {
    return new Result(
        null,
        diagnostics.all(),
        false
    );
}

Generated rules are verified:

for (final Value.RuleV rule
        : output.expandedRules()) {
    verifier.verifyRule(rule);
}

But runtime automata are lowered from automaton membership only:

for (final Map.Entry<
        String,
        List<Value.RuleV>> entry
        : output.automata().entrySet()) {
    // Only these members become RulePlan values.
}
Complexity
Input evaluation
Depends on the input expression; O(1) when reading an already evaluated matrix binding.
Iteration
O(N × T), where N is input count and T is transform application cost.
Layout-key check
Expected O(1) per generated layout.
Generated artifact storage
O(N), excluding aggregate values owned by each artifact.
Verification
O(N × V), where V is one artifact verification cost.
Runtime cost
None for expansion itself; it does not survive as an executable declaration.

Example

schema PolicyRow {
    Text key;
    Rule rule;
    Text reason;
    Measure measure;
    Duration span;
}

matrix<PolicyRow> policies {
    row(
        "chat-flood",
        P.2.1,
        "Repeated message flooding",
        Measure::MUTE,
        3h
    );

    row(
        "griefing",
        P.3.2,
        "Destruction of player structures",
        Measure::SUSPENSION,
        7d
    );
}

transform<PolicyRow -> Layout> make_layout =
    (entry) -> Layout {
        .key = entry.key,
        .rule = entry.rule,
        .reason = entry.reason,
        .measure = entry.measure,
        .duration = Duration::Fixed(entry.span),
        .escalation = Escalation::None,
        .annotations = {
            Annotation::Notify("staff"),
        },
    };

expand make_layout(policies);
