transform

keyword

Declares a named pure unary function, commonly used to convert a schema row into a generated layout or rule.

CategoryTop-level value declaration
IntroducesOne global function value
ArityExactly one parameter
Declared effectpure
Type identityStructural (Source) -> Target
Declaration evaluationAt compile time
ApplicationCompile time or runtime when reachable from a runtime closure
Java representationUsually Value.ClosureV, optionally Value.ComposeV

Syntax

transform<source-type -> target-type>
    name = function-expression;
transform-declaration =
    "transform" "<"
        source-type "->" target-type
    ">"
    identifier "=" expression ";" ;

Parameters

PartType or formDescription
source-type Resolvable value type Exact type of the transform's single argument.
target-type Resolvable value type Exact result type produced by successful application.
name Identifier Global value name used by calls, composition, and expansion.
function-expression (Source) -> Target with pure effect Usually a one-parameter lambda, but any compatible function-valued expression is accepted.

Constraints

Ill-formed if
  1. The source type is unknown at the declaration point.
  2. The target type is unknown at the declaration point.
  3. The transform name duplicates an earlier top-level declaration.
  4. The body does not have the structural type (Source) -> Target.
  5. The body lambda declares other than exactly one parameter.
  6. The function body calls a query-capable function.
  7. A composed inner function does not produce a value accepted by the outer function.
  • There is no implicit conversion between source, intermediate, or target types.
  • The source and target types may be primitive, records, lists, Measure, Sanction, Layout, or structural Rule<...> shapes where those names are accepted by source type resolution.
  • A transform is always unary even when its source type is itself a record or list.
  • A transform may refer only to earlier visible values and functions.
  • Recursive transform definitions are not representable under declaration-before-use.

Source type availability

The transform header uses the ordinary type namespace. Constructor path roots are not automatically type names.

For example, Layout, Sanction, Measure, primitive types, earlier schemas, list types, and structural rule shapes are valid header types. Other built-in sums may be constructible through paths without being accepted as general header types by the current implementation.

Semantics

Declaration evaluation

The transform declaration evaluates its function expression once while the unit is compiled and binds the resulting function value in the global environment.

For an ordinary lambda, declaration evaluation creates a closure. It does not execute the lambda body.

transform<Int -> Int> double =
    (value) -> value * 2;

Loading the declaration creates a closure over value * 2. Multiplication occurs only when double is applied.

Application

Applying the transform:

  1. evaluates the argument;
  2. creates a child lexical environment;
  3. binds the one declared parameter to the argument value;
  4. evaluates the transform body under the pure function effect;
  5. returns the resulting value.

Argument evaluation follows the allowance of the surrounding call site. The transform body itself remains pure.

Compile-time and runtime use

The declaration is always evaluated at compile time. Applications are not restricted to expand.

Effect preservation

A transform remains pure when called at runtime. Runtime availability does not grant it access to precedent, fingerprint_score, is_online, or now.

Closure capture

A lambda transform captures the evaluator environment object existing at declaration time. Earlier constants, matrices, and transforms can be resolved through that environment when the body later runs.

The root environment continues growing as later declarations are evaluated, but elaboration prevents the transform syntax from resolving a later declaration. Correct source programs therefore cannot use later growth as a forward-reference mechanism.

Composition

Member syntax denotes right-to-left function composition when the receiver has a pure unary function type and the member name resolves to another compatible earlier pure unary function.

outer.inner(argument)
=
outer(inner(argument))

Given:

transform<Int -> Int> double =
    (value) -> value * 2;

transform<Int -> Text> render =
    (value) -> match(value) {
        0 => "zero",
        _ => "non-zero",
    };

The composition render.double has type (Int) -> Text and applies double before render.

Composition lookup

The member after the dot is resolved as a global earlier transform or a global constant whose type is a pure unary function. It is not looked up as an arbitrary field on the receiver.

The same surface syntax denotes field access when the receiver is a record or event. Elaboration selects field access or composition from the receiver's static type.

Use by expand

expand imposes a narrower contract than an ordinary transform call:

  • the transform must be pure and unary;
  • the expansion argument must be a list of the transform source type;
  • the target type must be Layout or structural Rule<Events...>;
  • the transform is applied once per list element in source order.

A transform producing any other target type remains valid but cannot be the direct expansion target.

Evaluation failure

Transform application may fail during checked arithmetic, duration processing, text-pattern compilation, nested template instantiation, or evaluation-depth enforcement.

During compilation, the failure is reported against the relevant expression and rejects final publication of the unit. During runtime guard or verdict interpretation, the failure is contained by the dispatch engine.

No partial unit

An expansion may have temporarily appended artifacts produced by earlier rows before a later transform application fails. The planner still publishes no unit because the shared diagnostics contain an error.

Effect

pure The declared function type is always pure.

A pure transform may be called from both pure and query-capable positions. A query function cannot be assigned to a transform declaration.

Implementation notes, non-normative

The header creates one structural function type:

final Type source =
    resolveType(transform.from());

final Type target =
    resolveType(transform.to());

final Func functionType =
    new Func(
        List.of(source),
        target,
        Effect.PURE
    );

define(
    transform.name(),
    new Symbol.TransformSym(
        transform.name(),
        functionType,
        transform.body(),
        transform.pos()
    ),
    transform.pos()
);

check(
    transform.body(),
    functionType,
    new Scope(null),
    Effect.PURE
);

Evaluation binds the function value rather than executing an ordinary lambda body:

case Decl.Transform transform ->
    global.bind(
        transform.name(),
        eval(transform.body(), global)
    );

A lambda evaluates to a closure carrying its lexical environment and elaborated type:

case Expr.Lambda lambda ->
    new Value.ClosureV(
        lambda.params(),
        lambda.body(),
        env,
        (Type.Func) tables.typeOf.get(lambda)
    );

Application creates one child frame:

case Value.ClosureV closure -> {
    final Env frame =
        closure.env().child(args.size());

    for (int i = 0;
            i < closure.params().size();
            i++) {
        frame.bind(
            closure.params().get(i),
            args.get(i)
        );
    }

    yield eval(
        closure.body(),
        frame
    );
}

Composition stores outer and inner values explicitly:

record ComposeV(
    Value outer,
    Value inner
) implements Value {
}

Application is right-to-left:

case Value.ComposeV composition ->
    apply(
        composition.outer(),
        List.of(
            apply(
                composition.inner(),
                args,
                pos
            )
        ),
        pos
    );

Elaboration validates composition compatibility:

if (!assignable(
        inner.result(),
        outer.params().get(0))) {
    error(
        "E016",
        member.pos(),
        "composition mismatch: '"
            + member.name()
            + "' produces "
            + show(inner.result())
            + " but the receiver consumes "
            + show(outer.params().get(0))
    );

    throw Abort.INSTANCE;
}
  • Compile-time application depth is limited to 512.
  • Runtime interpreter application depth is limited to 256.
  • Compile-time closures capture the mutable root environment object, not a copied snapshot.
  • Name resolution tables prevent valid source from exploiting later root-environment growth.
  • Runtime interpretation reuses the closure's captured environment and creates only a small child frame per call.
  • Function values compare by identity rather than structural body equality.
Complexity
Declaration
O(1) closure construction, excluding elaboration of the body syntax.
Application
O(P + B), where P is parameter count, fixed at one for transforms, and B is body cost.
Composition
O(N + ΣB), where N is composed function count and B is each body cost.
Expansion
O(R × B), where R is input row count.
Call storage
One child environment and result value per active application.
Closure storage
One parameter list, body reference, environment reference, and function type.

Example

schema PolicyRow {
    Text key;
    Rule rule;
    Text reason;
    Measure measure;
    Duration span;
}

transform<PolicyRow -> Layout> to_layout =
    (entry) -> Layout {
        .key = entry.key,
        .rule = entry.rule,
        .reason = entry.reason,
        .measure = entry.measure,
        .duration = Duration::Fixed(entry.span),
        .escalation = Escalation::Steps({
            Duration::Fixed(entry.span),
            Duration::Fixed(entry.span * 4),
            Duration::Permanent,
        }),
        .annotations = {
            Annotation::Escalate,
            Annotation::Notify("staff"),
        },
    };

expand to_layout(policy_rows);
