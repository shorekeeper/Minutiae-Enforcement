match

keyword

Evaluates one subject and returns the body of the first matching ordered arm, with static subject, pattern, reachability, and exhaustiveness checks.

CategoryValue expression
Subject typesInt or a built-in sum type
Pattern formsExact integer, inclusive integer range, nullary constructor, wildcard
Arm orderNormative, first match wins
ExhaustivenessRequired
Result typeOne common arm-body type
Runtime stateNone

Syntax

match(subject) {
    pattern => result,
    ...
}
match-expression =
    "match" "(" expression ")" "{"
        match-arm
        { "," match-arm }
        [ "," ]
    "}" ;

match-arm =
    pattern "=>" expression ;

pattern =
      integer-literal
    | integer-literal ".." integer-literal
    | constructor-path
    | "_" ;

Constraints

Ill-formed if
  1. The subject is neither Int nor a sum type.
  2. An integer subject has no wildcard arm.
  3. A sum subject has neither a wildcard nor complete coverage of all its nullary constructors.
  4. An integer pattern is used with a sum subject.
  5. A constructor pattern belongs to another sum.
  6. A constructor pattern names a constructor with fields.
  7. A range has lower > upper.
  8. An exact integer pattern is duplicated.
  9. A constructor pattern is duplicated.
  10. An arm follows a wildcard arm.
  11. Arm result expressions do not have one common type.

Integer exhaustiveness

An Int subject always requires a wildcard. The elaborator does not attempt interval-union analysis over the signed 64-bit domain.

Sum exhaustiveness

Without a wildcard, every constructor of the sum must be nullary and named exactly once. A sum containing any constructor with fields therefore requires a wildcard under the current pattern language.

Overlapping ranges

The compiler validates inverted ranges but does not detect overlapping ranges, exact values covered by earlier ranges, or ranges fully shadowed by earlier ranges.

match(value) {
    0..10 => "first",
    5..15 => "second",
    7 => "third",
    _ => "other",
}

For subject 7, the first arm wins. The second and third arms remain accepted even though they cannot win for that value.

Ordering rule

Arms are tested in source order. ALAM defines no pattern specificity ordering and performs no automatic reordering.

Semantics

  1. Evaluate the subject exactly once.
  2. Test each pattern in source order.
  3. On the first match, evaluate that arm body.
  4. Return the resulting value.
  5. Do not evaluate later patterns or arm bodies.

Integer patterns

exact:
    subject == value

range:
    lower <= subject
    && subject <= upper

Constructor patterns

A constructor pattern compares the active constructor name. It does not destructure fields and introduces no local bindings.

Pattern paths must contain the sum root and constructor name. Longer arbitrary paths are not valid constructor patterns.

Wildcard

The wildcard matches every value. It binds no name and must be the final arm because every following arm is unreachable.

Result typing

When no expected result type is supplied, the first arm body determines the inferred result type and every later body is checked against it. When the surrounding position supplies an expected type, every arm body is checked directly against that type.

Conditional effects

The subject is always evaluated. Only the selected arm body is evaluated at runtime. All arm bodies are nevertheless type-checked and effect-checked statically.

A query call in an unselected arm is still rejected when the complete match occurs in a pure position.

Implementation notes, non-normative

Elaboration tracks exact integer and constructor duplicates:

final Set<Long> integers =
    new HashSet<>();

final Set<String> constructors =
    new HashSet<>();

Wildcard reachability is checked with one monotonic flag:

if (wildcardSeen) {
    error(
        "E012",
        arm.pos(),
        "arm is unreachable: "
            + "it follows a wildcard arm"
    );

    throw Abort.INSTANCE;
}

Range validation checks ordering only:

if (range.lo() > range.hi()) {
    error(
        "E013",
        range.pos(),
        "range bounds are inverted: "
            + range.lo()
            + " > "
            + range.hi()
    );

    throw Abort.INSTANCE;
}

Runtime and compile-time selection use the same first-match scan:

final Value subject =
    eval(match.subject(), env);

for (final Expr.Arm arm : match.arms()) {
    if (matches(
            arm.pattern(),
            subject)) {
        return eval(arm.body(), env);
    }
}

Pattern matching itself is a closed Java switch:

return switch (pattern) {
    case Pattern.WildPat ignored ->
        true;

    case Pattern.IntPat exact ->
        value instanceof Value.IntV integer
        && integer.value() == exact.value();

    case Pattern.RangePat range ->
        value instanceof Value.IntV integer
        && integer.value() >= range.lo()
        && integer.value() <= range.hi();

    case Pattern.PathPat path ->
        value instanceof Value.VariantV variant
        && variant.ctor().equals(
            path.segments().get(1)
        );
};
  • Constructor runtime matching compares constructor names after elaboration has established the sum type.
  • Ranges are stored as signed Java long bounds.
  • No interval tree or jump table is produced.
  • Reaching the end without a match is treated as an internal inconsistency and produces V013 at compile time or a contained runtime failure.
Complexity
Elaboration
Expected O(A + C), where A is arm count and C is total arm-body checking cost.
Evaluation
O(S + B), where S is patterns tested before the first match and B is selected body cost.
Pattern test
O(1) for every current pattern form.
Storage
O(A) AST arms; no additional runtime dispatch table.

Example

template<Measure M>
verdict by_measure = (event) -> Sanction {
    .target = event.subject,
    .cite = { P.2.1 },
    .measure = M,
    .duration = match(M) {
        Measure::MUTE =>
            Duration::Fixed(3h),

        Measure::SUSPENSION =>
            Duration::Fixed(7d),

        _ =>
            Duration::Permanent,
    },
    .escalation = Escalation::None,
    .annotations = {},
    .attribution = Attribution::System,
    .dry_run = DryRun::InheritSafety,
};
