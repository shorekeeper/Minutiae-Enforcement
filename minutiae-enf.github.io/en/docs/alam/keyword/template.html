template

keyword

Declares a parameterized compile-time producer of a guard or verdict value.

CategoryDeclaration keyword
IntroducesOne global template name
PhasesParsing, elaboration, compile-time evaluation, optional runtime instantiation
Declaration effectpure
Produced effectquery for guards, pure for verdicts
Runtime stateMemoization entries for distinct argument vectors

Syntax

template <parameter-list> [where-clause]
    template-kind name = expression;                         (1)

template <> template-kind name
    <specialization-argument-list> = expression;              (2)
  • (1)Declares a primary template.
  • (2)Declares an ordered specialization of an earlier primary template.

Grammar

parameter-list =
    template-parameter
    { "," template-parameter } ;

template-parameter =
    type identifier ;

template-kind =
      "guard"
    | "verdict" ;

specialization-argument-list =
    specialization-argument
    { "," specialization-argument } ;

specialization-argument =
      expression
    | "_" ;

Parameters

NameAdmissible type or formMeaning
Template parameter Measure, Duration, Int, Text, Rule Compile-time value bound for one instantiation.
Primary body function Fallback body when no specialization matches.
Fixed specialization argument Corresponding parameter type Matches by structural value equality.
_ Wildcard Matches every value in the corresponding parameter position.

Constraints

Ill-formed if
  1. A primary template has no parameter.
  2. A parameter type is not one of the five admissible types.
  3. A parameter name is duplicated.
  4. A specialization names no earlier primary template.
  5. The specialization kind differs from the primary kind.
  6. The specialization arity differs from the primary arity.
  7. A fixed specialization argument has the wrong type.
  8. A where expression is not a pure Bool.
  9. The primary or specialized body has an incompatible function type.

where

where {
    constraint;
    ...
}
where-clause =
    "where" "{"
        expression ";"
        { expression ";" }
    "}" ;

The block contains one or more ordered constraints. Every constraint is checked as a pure Bool expression and belongs to the primary template.

Postcondition

The selected body is evaluated only after every primary constraint has evaluated to true. Selecting a specialization does not bypass primary constraints.

Semantics

For an instantiation with arguments a1 through an:

  1. Evaluate the arguments from left to right.
  2. Form a structural argument key from their values.
  3. If the key is present in the instantiation cache, return the cached value.
  4. Create a child lexical environment and bind every primary parameter.
  5. Evaluate every primary constraint in source order.
  6. Test specializations in declaration order.
  7. For each specialization, compare fixed arguments by structural equality and accept wildcard arguments unconditionally.
  8. Select the first matching specialization. If none matches, select the primary body.
  9. Evaluate the selected body with all primary parameters bound.
  10. Store the result under the structural key and return it.
Ordering rule

Specializations are ordered. ALAM defines no most-specific-specialization relation. An earlier broad specialization may shadow a later narrow specialization.

Implementation notes, non-normative
  • Elaborator stores the primary as Symbol.TemplateSym.
  • Specializations are retained in declaration order in an ArrayList.
  • Compile-time memoization uses nested HashMap instances.
  • Runtime memoization uses nested ConcurrentHashMap instances because dispatch may occur on the main thread and asynchronous chat threads.
  • Value.key constructs the structural argument key.
  • Compile-time application depth is bounded at 512. Runtime application depth is bounded at 256.
Complexity
Cached
Expected O(A), where A is the number and structural size of arguments used to form the key.
Uncached
O(A + C + S × P + B), where C is constraint cost, S specialization count, P parameter count, and B selected-body cost.
Storage
O(K), where K is the number and retained size of distinct argument vectors.

A query built-in may be called inside a guard lambda. It may not be called while evaluating an argument, constraint, or verdict body.

Example

template<Measure M, Duration D>
where {
    D > 0s;
    D <= 30d;
}
verdict ladder = (event) -> Sanction {
    .target = event.subject,
    .cite = { P.2.1 },
    .measure = M,
    .duration = Duration::Fixed(D),
    .escalation = Escalation::None,
    .annotations = { Annotation::Notify("staff") },
    .attribution = Attribution::System,
    .dry_run = DryRun::InheritSafety,
};

template<> verdict ladder<Measure::CUSTODY, _> =
    (event) -> Sanction {
        .target = event.subject,
        .cite = { P.2.1 },
        .measure = Measure::CUSTODY,
        .duration = Duration::Permanent,
        .escalation = Escalation::None,
        .annotations = {},
        .attribution = Attribution::System,
        .dry_run = DryRun::InheritSafety,
    };
