MEMinutiae Enforcement docs

rule

keyword

Declares one named event-driven implication consisting of a compile-time trigger value, a runtime guard, and a pure runtime verdict.

CategoryTop-level declaration
IntroducesOne global explicit rule name
Structural typeRule<Event1, ..., EventN>
Trigger evaluationCompile time
Guard evaluationRuntime, query
Verdict evaluationRuntime after a passing guard, pure
Runtime publicationOnly through membership in an automaton
Failure postureGuard failure means false; verdict failure suppresses firing

Syntax

rule<event-list> name {
    condition {
        trigger = trigger-expression;
        guard = guard-expression;
    }
    => verdict = verdict-expression;
}
rule-declaration =
    "rule" "<"
        event-type
        { "," event-type }
    ">"
    identifier "{"
        condition-block
        "=>" verdict-clause
    "}" ;

condition-block =
    "condition" "{"
        trigger-clause
        guard-clause
    "}" ;

trigger-clause =
    "trigger" "=" expression ";" ;

guard-clause =
    "guard" "=" expression ";" ;

verdict-clause =
    "verdict" "=" expression ";" ;

Parameters

PartRequired typeDescription
Event list One or more known concrete event types Defines the ordered runtime argument vector and structural rule identity.
Name Identifier Global source name used by automaton membership.
trigger Trigger Pure compile-time firing precondition reduced to runtime plan data.
guard (Event1, ..., EventN) -> Bool Query-capable runtime condition over current or captured events.
verdict (Event1, ..., EventN) -> Sanction Pure runtime function producing the complete sanction descriptor.

Constraints

Ill-formed if
  1. The event list is empty.
  2. An event name is not a known concrete event type.
  3. The rule name duplicates any earlier top-level declaration.
  4. The trigger expression does not have type Trigger.
  5. The trigger shape is incompatible with the declared event list.
  6. The guard does not accept exactly the declared events in order and return Bool.
  7. The verdict does not accept exactly the declared events in order and return Sanction.
  8. The trigger or verdict requires a query-capable operation.
  9. A reachable external rule identifier is absent from the authoritative rule registry.

Trigger correspondence

TriggerDeclared event requirement
Trigger::AtomicExactly one event.
Trigger::RepeatedExactly one event.
Trigger::SequenceAt least two events, one matching step per event in the same order.

External rule identifiers

The ALAM rule name and an external rule identifier such as P.2.1 are separate concepts:

ValuePurpose
mute_floodNames the ALAM executable rule for automaton membership and reporting.
P.2.1Names the authoritative moderation rule cited by the resulting sanction.

Verification walks reachable rule identifier literals in the guard, verdict, and sequence step guards. Unknown identifiers produce R001.

Semantics

condition

The condition block groups one trigger clause and one rule guard clause. It does not create a separately typed value.

Source order is fixed: trigger first, guard second. Both are mandatory.

trigger

The trigger expression is evaluated once during compile-time normalization. The runtime does not reconstruct or re-evaluate the source expression for each event.

Evaluation produces a Value.VariantV whose constructor is validated against the rule's ordered event list. Lowering translates it to one sealed TriggerPlan form.

Trigger purity

The trigger may depend on literals, earlier constants, pure transforms, matrices, and pure template instantiations. It cannot depend on current precedent, online state, fingerprint score, or wall-clock time.

guard

The guard is a function over one argument per declared event. The exact time at which it runs depends on the trigger form.

TriggerRule guard timingArguments
Trigger::Atomic For every candidate event, before throttle acquisition. The current event record.
Trigger::Repeated For every candidate event, before the occurrence enters the window. The current event record.
Trigger::Sequence Once after all steps complete and captured facts are drained. One captured event record per declared event.

A false atomic or repeated guard leaves no trigger-state change. A false sequence rule guard occurs after the completed sequence has already been consumed and reset.

Guard failure

If runtime interpretation fails, the guard is treated as unsatisfied. The throwable does not propagate to the Paper event handler.

A guard failure may result from checked arithmetic, invalid dynamic regular expression, missing applicable value, template constraint failure, or evaluation-depth exhaustion.

Sequence step guards

Step guards are distinct from the final rule guard. Each step guard receives only the incoming event of that step and decides whether it may advance the partial match.

The final rule guard receives the complete captured vector and can enforce relationships between steps, such as equal subjects or correlated values.

verdict

The verdict executes only after:

  1. the automaton is not already self-muted;
  2. the trigger condition passes;
  3. the rule guard returns true;
  4. the automaton throttle admits the firing.

The verdict receives the same argument vector as the final rule guard and must return one total Sanction record.

Verdict purity

The verdict may read event fields and captured compile-time values. It cannot call query-capable built-ins. This prevents the sanction descriptor from observing a second mutable policy state after the guard has admitted the firing.

After the verdict returns, materialization and sanction resolution occur outside the language expression. They may perform cached queries, asynchronous storage operations, permission checks, and player effects.

Verdict failure

A verdict interpreter failure suppresses sanction issuance and writes an English operator warning containing the qualified ALAM rule name and failure message.

The throttle entry has already been acquired by this point and is not refunded. Trigger completion state has also already been consumed.

Runtime indexing

Each automaton membership becomes an ArmedRule. The dispatch engine indexes it under every event kind returned by the plan:

  • atomic and repeated rules are indexed under their single event;
  • sequence rules are indexed under every step event.

On one incoming event, every candidate in the corresponding index list is gated independently. The engine does not stop after the first firing.

Qualified name

Runtime reporting qualifies a rule by its automaton:

automaton-name "." rule-name

The same explicit rule used in two automata therefore has two runtime qualified names and two independent trigger partition namespaces.

Unused explicit rules

An explicit rule that belongs to no automaton is compiled and verified but never appears in the runtime dispatch engine. Verification emits W003.

Simulation

Simulation gates matching candidates without consuming window, sequence, throttle, capture, or self-mute state. A passing verdict is resolved through forced dry-run.

Sequence simulation has no real earlier captured events. It tests the final step and adapts the synthetic completing event under all event shapes for the final guard and verdict argument vector.

Implementation notes, non-normative

Elaboration creates the structural event shape and checks all three positions:

final List<Event> events =
    new ArrayList<>(rule.events().size());

for (final String name : rule.events()) {
    final Event event =
        Builtins.event(name);

    if (event == null) {
        error(
            "E025",
            rule.pos(),
            "unknown event type '" + name + "'"
        );

        throw Abort.INSTANCE;
    }

    events.add(event);
}

final RuleT shape =
    new RuleT(List.copyOf(events));

define(
    rule.name(),
    new Symbol.RuleSym(
        rule.name(),
        shape,
        rule,
        rule.pos()
    ),
    rule.pos()
);

check(
    rule.trigger(),
    Builtins.TRIGGER,
    emptyScope,
    Effect.PURE
);

check(
    rule.guard(),
    guardType(events),
    emptyScope,
    Effect.PURE
);

check(
    rule.verdict(),
    verdictType(events),
    emptyScope,
    Effect.PURE
);
Why guard is checked from a pure outer position

The guard expression constructs a function value under a pure declaration position. When the expression is a lambda, bidirectional checking grants the lambda body the QUERY effect carried by the expected function type.

Evaluation reduces the trigger and captures guard and verdict values:

final Value triggerValue =
    eval(rule.trigger(), global);

final Value.VariantV trigger =
    (Value.VariantV) triggerValue;

validateTrigger(
    trigger,
    events,
    rule.pos()
);

final Value guard =
    eval(rule.guard(), global);

final Value verdict =
    eval(rule.verdict(), global);

return new Value.RuleV(
    rule.name(),
    events,
    trigger,
    guard,
    verdict
);

Lowering produces one first-order plan:

return new RulePlan(
    rule.name(),
    eventKinds,
    eventShapes,
    lowerTrigger(rule.trigger()),
    rule.guard(),
    rule.verdict()
);

Runtime gate selection uses the sealed trigger hierarchy:

return switch (rule.plan().trigger()) {
    case TriggerPlan.Atomic ignored ->
        gateSingle(
            rule,
            facts,
            now,
            countFiring,
            null
        );

    case TriggerPlan.Repeated repeated ->
        gateSingle(
            rule,
            facts,
            now,
            countFiring,
            repeated
        );

    case TriggerPlan.Sequence sequence ->
        gateSequence(
            rule,
            sequence,
            facts,
            now,
            countFiring
        );
};

Guard failure is explicitly contained:

private Boolean guard(
        final ArmedRule rule,
        final Value guard,
        final List<Value> args) {
    try {
        return rule.interp()
            .applyGuard(
                guard,
                args,
                env
            );
    } catch (final Interp.Failure failure) {
        return null;
    }
}

Verdict failure is logged and suppressed:

final Value.RecordV sanction;

try {
    sanction = rule.interp()
        .applyVerdict(
            rule.plan().verdict(),
            args,
            env
        );
} catch (final Interp.Failure failure) {
    log.warn(
        "dispatch",
        "rule '%s' verdict failed: %s",
        rule.qualifiedName(),
        failure.getMessage()
    );

    return;
}
  • RulePlan stores arrays by reference and exposes them directly through record accessors; callers treat them as immutable.
  • Event record shapes are interned per event name during lowering of one unit.
  • Rule plans carry no original source positions.
  • The unit's shared Interp is the only interpreter context valid for its closures.
  • Automatic sanctions converge with manual sanctions at SanctionResolver.
  • A successful non-simulated firing appends an asynchronous dispatch audit entry after executor invocation is accepted.
Complexity
Elaboration
O(N + G + V + T), where N is event count and G, V, T are guard, verdict, and trigger syntax-check costs.
Lowering
O(N + S), where S is sequence step count.
Event-index construction
O(N) entries per rule membership.
Atomic dispatch
O(A + G + H + V), for adaptation, guard, throttle, and verdict.
Repeated dispatch
O(A + G + C + H + V) on a firing, where C is recurrence count scan.
Sequence completion
O(N + ΣGs + G + H + V), including step guards and captured-event adaptation.
Plan storage
O(N + S) per automaton membership, excluding shared closure environments.

Example

rule<Login, Chat> immediate_repeat_offense {
    condition {
        trigger = Trigger::Sequence {
            .within = 10m,
            .group_by = GroupBy::Subject,
            .steps = {
                Step<Login> {
                    .guard = Guard::Always,
                },
                Step<Chat> {
                    .guard = (chat) ->
                        chat.length > 120,
                },
            },
        };

        guard = (login, chat) ->
            login.subject == chat.subject
            && precedent(
                chat.subject,
                P.2.1
            ) > 0;
    }

    => verdict = (login, chat) -> Sanction {
        .target = chat.subject,
        .cite = { P.2.1 },
        .measure = Measure::MUTE,
        .duration = Duration::Fixed(1d),
        .escalation = Escalation::None,
        .annotations = {
            Annotation::Reason(
                "Repeated flooding after login"
            ),
            Annotation::Notify("staff"),
        },
        .attribution = Attribution::System,
        .dry_run = DryRun::InheritSafety,
    };
}
