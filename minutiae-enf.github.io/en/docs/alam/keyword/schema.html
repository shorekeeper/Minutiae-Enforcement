schema

keyword

Declares a nominal record type with an ordered list of named mandatory fields.

CategoryTop-level type declaration
IntroducesOne global schema type name
Value bindingNone
Field orderDeclaration order, semantically significant
Field optionalityNone; every field is mandatory at construction
Type identityNominal by schema name
Runtime representationType.Rec plus ordered Value.RecordV fields

Syntax

schema name {
    field-declarations
}
schema-declaration =
    "schema" identifier "{"
        { field-declaration }
    "}" ;

field-declaration =
    type field-name ";" ;

field-name =
      identifier
    | keyword ;

Fields

PartFormDescription
Schema name Identifier Nominal type identity entered into the global symbol table.
Field type Resolvable source type Exact type required of the corresponding value field.
Field name Identifier or keyword Member name used by construction, field access, with, matrix row ordering, and transform logic.

Constraints

Ill-formed if
  1. The schema name duplicates an earlier top-level declaration.
  2. The schema name collides with a built-in type-namespace name.
  3. A field name is duplicated within the schema.
  4. A field type is unknown at the declaration point.
  5. A field type is applied with unsupported type arguments.
  • A schema may contain zero fields.
  • Field names may use reserved keywords because the parser knows that only a field name can occur after a field type.
  • A schema may refer only to types already available at its declaration point.
  • Self-recursive and mutually recursive schemas are not representable under declaration-before-use.
  • Two schemas with identical fields remain distinct types when their names differ.
  • Reordering fields changes positional matrix semantics even when field names and types are otherwise unchanged.

Namespace detail

A value path such as Trigger::Atomic may exist even though Trigger is not accepted as a field type in a user schema. Type names and constructor path roots are resolved by separate tables.

Semantics

Nominal identity

A schema's name is its type identity. Field structure does not create structural equivalence.

schema Left {
    Int value;
}

schema Right {
    Int value;
}

Left and Right are not assignable despite having identical fields.

Field order

Fields retain source order. This order controls:

  • the positional meaning of each matrix cell;
  • the internal ordered storage of every record value;
  • the ordering used when a named record construction is normalized;
  • the index used by field access and with updates.
Compatibility consequence

Changing field order is a source compatibility change for every existing matrix over that schema. Matrix rows are positional and do not retain field labels.

Construction

A schema value is constructed by naming the schema and supplying every field:

TariffRow {
    .measure = Measure::MUTE,
    .span = 3h,
    .priority = 1,
}

Named initializers may appear in any source order. Evaluation reorders them to schema order before creating the record value.

An anonymous record literal may be used when the surrounding position supplies the expected schema type:

const TariffRow row = {
    .measure = Measure::MUTE,
    .span = 3h,
    .priority = 1,
};

Mandatory fields

Every declared field must be supplied. ALAM has no optional fields, default expressions, null value, field omission, or automatic zero initialization for record construction.

Neutral values exist only during platform event adaptation. They are not injected into user schema constructions.

Field access

For a value record of schema type, record.name returns the field value. The receiver's static nominal type determines whether the field exists.

The parser admits keywords as member names after a dot, so fields such as rule, guard, or with remain accessible.

Functional update

with creates another value of the same nominal schema. It copies the ordered field list and replaces selected indexes. It cannot add fields or change the schema type.

No value binding

A schema declaration contributes only a type symbol. It does not bind a value named after the schema in the ordinary evaluation environment.

The same source spelling is nevertheless recognized as a construction target by elaboration when followed by a record body.

Effect

A schema declaration contains no expression and has no runtime effect. Expressions used to construct schema values inherit the allowance of their surrounding position.

Implementation notes, non-normative

Elaboration builds the ordered field list and rejects duplicates:

final List<Field> fields =
    new ArrayList<>(schema.fields().size());

final Set<String> seen =
    new HashSet<>(8);

for (final Decl.Schema.Field field
        : schema.fields()) {
    if (!seen.add(field.name())) {
        error(
            "E014",
            field.pos(),
            "duplicate field '"
                + field.name()
                + "' in schema '"
                + schema.name()
                + "'"
        );

        throw Abort.INSTANCE;
    }

    fields.add(new Field(
        field.name(),
        resolveType(field.type())
    ));
}

Built-in type-name collisions are checked before global registration:

if (Builtins.typeNamed(
        schema.name()) != null) {
    error(
        "E014",
        schema.pos(),
        "schema name '"
            + schema.name()
            + "' collides with a built-in type"
    );

    throw Abort.INSTANCE;
}

The nominal type is one Java record:

final Rec recordType =
    new Rec(
        schema.name(),
        List.copyOf(fields)
    );

define(
    schema.name(),
    new Symbol.SchemaSym(
        schema.name(),
        recordType,
        schema.pos()
    ),
    schema.pos()
);

Named construction is normalized into schema order:

final Map<String, Value> byName =
    new HashMap<>(
        literal.fields().size() * 2
    );

for (final Expr.FieldInit field
        : literal.fields()) {
    byName.put(
        field.name(),
        eval(field.value(), env)
    );
}

final List<Value> ordered =
    new ArrayList<>(
        recordType.fields().size()
    );

for (final Type.Field field
        : recordType.fields()) {
    ordered.add(byName.get(field.name()));
}

return new Value.RecordV(
    recordType,
    List.copyOf(ordered)
);

Field lookup is currently linear in field count:

public int fieldIndex(
        final String fieldName) {
    for (int i = 0;
            i < fields.size();
            i++) {
        if (fields.get(i)
                .name()
                .equals(fieldName)) {
            return i;
        }
    }

    return -1;
}
  • Type.Rec and Value.RecordV are immutable after publication.
  • A record value stores a direct reference to its nominal type.
  • Aggregate values do not use Java reflection.
  • Field lookup does not currently maintain a name-to-index hash table.
  • Type identity compares record names, relying on global duplicate-name rejection for soundness.
Complexity
Schema elaboration
Expected O(F), where F is field count.
Named construction
Expected O(F) for value evaluation, temporary name mapping, and ordered materialization.
Field access
O(F) name lookup, then O(1) value-list indexing.
with update
O(F + U × F), where U is updated-field count under linear field lookup.
Value storage
O(F) references per record value, plus one shared type object.

Example

schema TariffRow {
    Text key;
    Rule rule;
    Measure measure;
    Duration span;
    Int priority;
}

const TariffRow default_tariff = TariffRow {
    .key = "chat-flood",
    .rule = P.2.1,
    .measure = Measure::MUTE,
    .span = 3h,
    .priority = 10,
};
