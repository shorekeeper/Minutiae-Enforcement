automaton

keyword

Declares a named ordered set of rules that shares one arming decision, firing throttle, and self-mute boundary at runtime.

CategoryTop-level declaration
IntroducesOne global automaton name
MembersOrdered references to earlier explicit rule declarations
InheritanceOptional, from one earlier automaton
Compile-time outputImmutable ordered list of residual rule values
Runtime boundaryArming, throttle, self-muting, reporting, and audit
Runtime reloadNot supported by the current boot wiring

Syntax

automaton name {
    automaton-items
}                                                            (1)

automaton name : parent {
    automaton-items
}                                                            (2)
  • (1)Declares a root automaton with initially empty membership.
  • (2)Declares an automaton initialized from the fully resolved membership of an earlier parent.
automaton-declaration =
    "automaton" identifier
    [ ":" parent-automaton ]
    "{"
        { automaton-item }
    "}" ;

automaton-item =
      "use" rule-name ";"
    | "override" "<" event-type ">"
      "use" rule-name ";" ;

Parameters

PartFormDescription
name Identifier Global automaton identity used by compilation, runtime arming, throttling, self-muting, reports, and audit.
parent Earlier automaton identifier Optional source of already resolved ordered membership.
use Earlier explicit rule name Appends one rule to the current membership.
override Event type and earlier explicit rule name Removes every current member whose principal event equals the selected event, then appends the replacement.

Constraints

Ill-formed if
  1. The automaton name duplicates any earlier top-level declaration name.
  2. The parent does not name an earlier automaton.
  3. A use item does not name an earlier explicit rule declaration.
  4. An override event is not a known concrete event.
  5. An override replacement does not name an earlier explicit rule declaration.
  6. The replacement rule's first declared event differs from the selected override event.
  7. A use item inserts a rule already present in current membership.
  • A root or derived automaton may contain no items.
  • Inheritance is single-parent and source-order only.
  • Forward references are not resolved after parsing.
  • An automaton may inherit from an empty automaton.
  • The same explicit rule may belong to several different automata.
  • Rules produced by expand are not entered into the global symbol table and cannot currently be named by use or override.
Generated-rule limitation

expand may produce residual rule values, but the current language has no membership syntax that addresses their synthesized names. Only explicit earlier rule declarations can be armed through an automaton.

Semantics

Initial membership

A root automaton starts with an empty ordered map. A derived automaton copies the parent's final resolved list into a new ordered map before applying its own items.

root:
    current = []

derived:
    current = copy(parent.resolved_members)

The parent is not referenced dynamically. Later operations on the child do not mutate the parent, and the runtime plan retains no parent pointer.

use

A use item appends the named rule to current membership. If the rule name is already present, evaluation produces V017.

Duplicate membership is tested after inheritance and after all earlier items. Therefore a child cannot repeat a rule already inherited from its parent without first removing it through an applicable override.

override

An override operates on the complete current membership at that point in source order. It is not limited to rules inherited from the parent.

  1. Inspect every current member.
  2. Read its first declared event type.
  3. Remove the member when that event equals the selected override event.
  4. Append the replacement rule.
Principal event

The principal event of a rule is its first declared event. For a sequence rule declared as rule<Login, Chat>, the principal event is Login. An override<Chat> item does not remove that rule.

An override removes every matching member, not merely the first one. One replacement may therefore replace several inherited or locally inserted rules.

If no member is removed, the replacement is still inserted and the compiler emits W001.

Membership order

Unaffected inherited and local members retain their relative order. Removed members disappear. The replacement is appended after every member that remains.

before:
    [chat_a, login_a, chat_b, break_a]

override<Chat> use chat_strict;

after:
    [login_a, break_a, chat_strict]

Membership order determines the order in which plans are published from the automaton. The dispatch engine preserves candidate-list order for each event index. If several rules fire on one event, their verdicts are therefore evaluated in deterministic publication order for a fixed set of accepted source files.

One rule in several automata

The same explicit rule may be used by multiple automata. Each membership becomes a separate ArmedRule associated with its owning automaton.

The rule body and interpreter context may be shared, but runtime safety state is scoped by automaton name:

  • arming is checked per automaton;
  • throttle timestamps are stored per automaton;
  • self-muting applies per automaton;
  • trigger partition keys include the automaton name;
  • audit and reports use the owning automaton name.

Arming

An automaton is live when global arming is enabled or its name appears in the configured armed set:

is_armed =
    armed_globally
    || armed_automata.contains(name)

An unarmed automaton is not skipped. Its rules still evaluate and produce forced dry-run sanctions.

Unarmed-state behavior

On live events, an unarmed automaton still consumes repeated windows, sequence progress, and throttle capacity before the firing is converted to dry-run. Only explicit simulation avoids consuming those runtime structures.

Throttle

All rules in one automaton share one per-minute throttle. A firing from any member consumes one entry from that automaton's timestamp ring.

When the trailing one-minute count reaches the configured limit, acquisition fails and the automaton enters the self-mute set.

Self-muting

A self-muted automaton rejects every candidate before trigger-specific gating. Repeated and sequence state is not advanced while the automaton remains muted.

The current engine exposes no runtime command that clears one self-muted automaton. The state lasts for the lifetime of the DispatchEngine instance.

Reload behavior

The current reload manager does not register AlamService or reconstruct DispatchEngine. Editing ALAM files or clearing self-mute state requires a plugin or server restart under the supplied boot wiring.

Publication

Each accepted source unit publishes automata in declaration order through a LinkedHashMap. Rules within each automaton are published in resolved membership order.

The service later flattens every unit and automaton into immutable ArmedRule values. A source file that fails any compiler phase contributes no automaton or rule plan.

Effect

pure Automaton declaration and membership resolution occur at compile time. Runtime member execution may perform query guards and sanction effects according to each rule.

Implementation notes, non-normative

The AST stores the declaration and its ordered items:

record Automaton(
    String name,
    String parent,
    List<Item> items,
    Pos pos
) implements Decl {
}

Evaluation copies inherited membership into an ordered map:

final LinkedHashMap<String, Value.RuleV> current =
    new LinkedHashMap<>();

if (a.parent() != null) {
    final List<Value.RuleV> parent =
        automata.get(a.parent());

    if (parent != null) {
        for (final Value.RuleV rule : parent) {
            current.put(rule.name(), rule);
        }
    }
}

use explicitly rejects duplicate membership:

case Decl.Automaton.Use use -> {
    final Value.RuleV rule =
        rules.get(use.rule());

    if (current.putIfAbsent(
            use.rule(),
            rule) != null) {
        error(
            "V017",
            use.pos(),
            "rule '" + use.rule()
                + "' is already a member of automaton '"
                + a.name()
                + "'"
        );

        throw Abort.INSTANCE;
    }
}

override removes current members by first event and then appends its replacement:

case Decl.Automaton.Override override -> {
    int removed = 0;
    final var iterator =
        current.entrySet().iterator();

    while (iterator.hasNext()) {
        if (iterator.next()
                .getValue()
                .events()
                .get(0)
                .name()
                .equals(override.event())) {
            iterator.remove();
            removed++;
        }
    }

    if (removed == 0) {
        diags.warning(
            "W001",
            override.pos().line(),
            override.pos().column(),
            "override on '"
                + override.event()
                + "' removed no inherited rule"
        );
    }

    current.put(
        override.rule(),
        rules.get(override.rule())
    );
}
Diagnostic wording

The W001 log text says “removed no inherited rule”, but the implementation scans all current members, including rules inserted earlier in the same child automaton.

Runtime flattening creates one armed membership value:

out.add(new ArmedRule(
    automatonName,
    plan,
    unit.interp()
));

Throttle and self-mute state are keyed by automaton name:

if (throttle.tryAcquire(
        rule.automaton(),
        env.now())) {
    return true;
}

muted.add(rule.automaton());
return false;
  • Throttle uses a Java HashMap and is not internally synchronized.
  • Its documented implementation assumption is confinement to one dispatch path, although live chat and main-thread events may call the engine from different threads.
  • Repeated and sequence trackers use concurrent maps, but some per-partition mutable values also rely on serialized advancement.
  • Cross-thread automata containing both asynchronous Chat rules and main-thread event rules should be reviewed carefully under the current runtime model.

Example

automaton base_moderation {
    use chat_flood;
    use suspicious_login;
    use container_break;
}

automaton strict_chat : base_moderation {
    override<Chat> use strict_chat_policy;
    use post_login_sequence;
}

The derived automaton:

  1. copies all three parent members;
  2. removes every current member whose first event is Chat;
  3. appends strict_chat_policy;
  4. appends post_login_sequence if it is not already present.
