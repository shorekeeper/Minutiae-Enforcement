Chat

event

An observed player chat message represented as plain text with its UTF-16 code-unit length.

CategoryConcrete event
BaseEvent
Paper sourceAsyncChatEvent
Listener priorityMONITOR
Cancelled eventsIgnored

Syntax

rule<Chat> name { ... }                                  (1)

Step<Chat> {
    .guard = chat_guard,
}                                                          (2)
  • (1)Declares a rule whose one event argument has type Chat.
  • (2)Declares a sequence step admitted by a guard over one Chat value.

Fields

FieldTypeSourceDescription
subject Account Speaking player's UUID Stable identity of the player who sent the message.
subject_name Text Speaking player's current name Display name captured when dispatch facts are created.
message Text Plain serialization of the Adventure message component Unformatted message content presented to ALAM guards and verdicts.
length Int message.length() Java string length of the serialized message.

Constraints

  • Chat may be used only where a known concrete event type is required.
  • A rule using Trigger::Atomic or Trigger::Repeated must declare only this one event.
  • A Step<Chat> guard accepts exactly one Chat parameter and returns Bool.
  • The length field is an Int. It is not implicitly converted to Real.
  • The event is dispatched only when Paper has not already cancelled the chat event before the monitor listener runs.

Semantics

The dispatch listener observes the final message component at Paper monitor priority. It serializes that component with PlainTextComponentSerializer and constructs one EventFacts value.

The listener itself never cancels the chat event and does not modify its viewer set. Enforcement effects are expressed through sanctions issued by the dispatch engine rather than by mutating the triggering message.

message

The message field contains plain text. Adventure colors, click events, hover events, font data, and other component structure are not exposed to ALAM.

The plain serializer is not the chat-history sanitizer. The ALAM listener does not remove ampersands or section-sign characters and does not truncate the message to the chat-history retention limit.

length

The length is computed with Java String.length(). It counts UTF-16 code units, not Unicode code points, grapheme clusters, rendered glyphs, or bytes.

A supplementary Unicode character represented by one surrogate pair contributes two to length.

Observation timing

The event is observed at monitor priority with ignoreCancelled = true. A lower-priority plugin may cancel the message before ALAM sees it. A later monitor handler should not modify the event by Paper convention, but ALAM cannot enforce behavior of other plugins.

Threading

The listener runs on Paper's asynchronous chat thread. Guard evaluation, repeated-window updates, sequence advancement, verdict interpretation, and initial sanction resolution may therefore begin off the main server thread.

Query built-ins must not perform blocking I/O on this path. Storage-backed guard data is served from bounded caches, with asynchronous fills scheduled on a miss.

Independent chat subsystems

The plugin also registers behavioral mute enforcement and chat transcript capture listeners. These are separate from ALAM dispatch:

  • behavioral mute enforcement may cancel the chat before the ALAM monitor listener;
  • shadow behavior may reduce the message viewer set;
  • chat-history capture serializes and sanitizes a bounded copy for possible transcript persistence;
  • ALAM dispatch receives its own plain serialization and does not read the transcript ring.
Implementation notes, non-normative

The built-in event shape is:

"Chat", new Event(
    "Chat",
    List.of(
        new Field("subject", ACCOUNT),
        new Field("subject_name", TEXT),
        new Field("message", TEXT),
        new Field("length", INT)
    )
)

The live listener creates facts directly from the Paper event:

@EventHandler(
    priority = EventPriority.MONITOR,
    ignoreCancelled = true
)
public void onChat(final AsyncChatEvent event) {
    final Player player = event.getPlayer();
    final String message =
        plain.serialize(event.message());

    engine.handle(new EventFacts(
        EventKind.CHAT,
        player.getUniqueId(),
        player.getName(),
        Map.of(
            "message", message,
            "length", (long) message.length()
        )
    ));
}
  • length is explicitly widened from Java int to long.
  • The fact map is immutable because it is created with Map.of.
  • EventKind.CHAT is recognized from the source event name case-insensitively during runtime command simulation.
  • The dispatch listener imposes no explicit message-length bound.
Complexity
Serialization
O(M), where M is the Adventure component's serialized plain-text size.
Length
O(1) for Java String.length().
Fact construction
O(1), excluding retained message text.
Rule dispatch
O(R), plus trigger and interpreter cost for R candidate rules indexed under Chat.

Example

rule<Chat> long_repeated_message {
    condition {
        trigger = Trigger::Repeated {
            .count = 5,
            .within = 30s,
            .group_by = GroupBy::Subject,
        };
        guard = (event) ->
            event.length > 120
            && event.message ~ "[A-Z]{20,}";
    }
    => verdict = chat_verdict<Measure::MUTE, 3h>;
}
