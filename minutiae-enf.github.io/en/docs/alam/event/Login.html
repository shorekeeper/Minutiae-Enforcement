Login

event

A successful player join observed after admission, containing the connecting player's account, display name, and available remote address text.

CategoryConcrete event
BaseEvent
Paper sourcePlayerJoinEvent
Listener priorityMONITOR
Dispatch threadPaper main server thread
Admission pointAfter successful pre-login and player join
Live listenerRegistered by the dispatch boot stage

Syntax

rule<Login> name { ... }                                 (1)

Step<Login> {
    .guard = login_guard,
}                                                          (2)
  • (1)Declares a rule observing successful player joins.
  • (2)Declares one successful-login step of a sequence.

Fields

FieldTypeSourceDescription
subject Account Joined player's UUID Stable identity of the player who completed login.
subject_name Text Joined player's current name Display name captured during the join event.
ip Text Remote socket address when available Text returned by InetAddress.getHostAddress(), or empty text after adaptation when unavailable.

Constraints

  • Login represents a successful join, not an attempted or rejected connection.
  • A login rule cannot prevent the same join event from occurring because dispatch begins after the player has joined.
  • The ip field may be empty when Paper exposes no socket address.
  • The field contains address text only. It exposes no port, hostname, provider, proxy classification, or fingerprint evidence decomposition.
  • A Step<Login> guard accepts one Login value and returns Bool.

Semantics

The listener runs for PlayerJoinEvent. The access guard and fingerprint pre-login listener have already run by this point. A connection denied by either pre-login path never produces a live Login event.

Post-join timing

A sanction fired from Login is issued after the subject exists as an online Paper player. A live connection-blocking sanction may therefore disconnect the player shortly after join rather than reject the original connection before world entry.

Behavioral sanctions may be applied to the now-online player through the executor's main-thread effect path.

Admission distinction

Use the existing access guard for persisted active bans. A Login rule reacts to a successful join and is not a synchronous pre-login admission policy.

ip

When both the player's socket address and underlying network address are available, the listener stores getHostAddress(). This normally produces a numeric IPv4 or IPv6 string without DNS lookup.

When unavailable, the raw fact map contains no ip key. EventAdapter then supplies empty text because the declared field type is Text.

Online state

During ordinary join dispatch, is_online(event.subject) normally returns true because the player has already joined. Definitions should not use that expression as the sole distinction between admitted and rejected login attempts.

Fingerprint distinction

The fingerprint subsystem performs its own assessment during AsyncPlayerPreLoginEvent. The Login event exposes only the IP text and does not automatically include the computed fingerprint score.

A guard may call fingerprint_score(event.subject), but that query reads a separate cache. On a cache miss it returns 0.0 and schedules an asynchronous fill, so the first login evaluation may observe the neutral score.

Other join listeners

The plugin also uses the join event to load persisted behavioral state, reconcile concealment, enrich fingerprint sessions, record session intervals, update names, and load staff preferences. These listeners are independent of ALAM dispatch and may complete asynchronously.

Implementation notes, non-normative

The built-in shape is:

"Login", new Event(
    "Login",
    List.of(
        new Field("subject", ACCOUNT),
        new Field("subject_name", TEXT),
        new Field("ip", TEXT)
    )
)

The listener omits the fact rather than storing a null value:

@EventHandler(priority = EventPriority.MONITOR)
public void onJoin(final PlayerJoinEvent event) {
    final Map<String, Object> facts =
        new HashMap<>(2);

    if (event.getPlayer().getAddress() != null
            && event.getPlayer()
                .getAddress()
                .getAddress() != null) {
        facts.put(
            "ip",
            event.getPlayer()
                .getAddress()
                .getAddress()
                .getHostAddress()
        );
    }

    engine.handle(new EventFacts(
        EventKind.LOGIN,
        event.getPlayer().getUniqueId(),
        event.getPlayer().getName(),
        facts
    ));
}
  • The fact map is mutable during construction but is not modified after dispatch begins.
  • The listener performs no reverse-DNS lookup.
  • Simulation defaults an empty supplied IP to 127.0.0.1, which differs from the live missing-address fallback of empty text.
  • A live join rule is indexed under EventKind.LOGIN.
Complexity
Address extraction
O(L), where L is the numeric address text length.
Fact construction
Expected O(1).
Rule dispatch
O(R), plus trigger and interpreter cost for R rules indexed under Login.
DNS or storage
No direct operation in the listener.

Example

rule<Login> suspicious_network {
    condition {
        trigger = Trigger::Atomic;
        guard = (event) ->
            event.ip ~ "^203\\.0\\.113\\."
            && precedent(event.subject, P.5.1) > 0;
    }
    => verdict =
        login_verdict<Measure::SUSPENSION, 7d>;
}
