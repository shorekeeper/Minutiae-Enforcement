Evasion

event

A scored suspected-evasion event supported by the ALAM type system, execution plans, dispatch index, and simulation surface.

CategoryConcrete event
BaseEvent
Fieldssubject, subject_name, score
SimulationSupported
Plan loweringSupported as EventKind.EVASION

Current live sourceNone in the supplied runtime wiring

Syntax

rule<Evasion> name { ... }                               (1)

Step<Evasion> {
    .guard = evasion_guard,
}                                                          (2)
  • (1)Declares a rule over one evasion-score record.
  • (2)Declares an evasion-score step in a sequence.

Fields

FieldTypeDescription
subject Account Account assessed as the suspected evader.
subject_name Text Display name captured by the producer of the event.
score Real Producer-supplied evasion score, conventionally a probability in the range 0 through 1.

Constraints

  • The compiler validates the score field as Real.
  • The ALAM type system does not enforce the conventional range 0 through 1.
  • A live or programmatic event producer is responsible for selecting the subject and score.
  • The current plugin boot sequence does not register an evasion dispatch listener.
  • A rule over Evasion may compile and appear armed while receiving no live event from the current built-in listener set.

Semantics

When an EventFacts value with kind EVASION reaches DispatchEngine.handle, it is processed in the same manner as other event kinds. Candidate rules are selected from the event index, facts are adapted to the Evasion record shape, and trigger gating proceeds normally.

score

The event field is the score supplied by the event producer. ALAM does not recompute fingerprint evidence while adapting the event.

A definition may compare the field directly:

event.score >= 0.90

The value should not be assumed equal to a later call to fingerprint_score(event.subject). The field is an event snapshot, while the function reads the current non-blocking cache.

Current live wiring

The fingerprint subsystem assesses login probes during AsyncPlayerPreLoginEvent and raises staff alerts when the posterior probability reaches its threshold. In the supplied implementation, that alert path does not construct EventFacts(EventKind.EVASION, ...) and does not call DispatchEngine.handle.

Current integration status

Evasion is a valid language and plan event, but built-in live fingerprint alerts are not connected to ALAM dispatch. The event currently requires simulation or an additional programmatic producer.

This distinction is intentional documentation of the current code, not a language restriction. Adding a producer does not require a grammar or plan change because EventKind.EVASION and the event shape already exist.

Simulation

The command surface accepts:

/alam simulate evasion subject score

The simulator parses the supplied data as a Java double. If parsing fails, it uses the default score 1.0.

Simulation forces sanction dry-run and does not consume trigger windows, sequence state, throttle capacity, or the self-mute set.

Programmatic production

An integration may dispatch the event by constructing platform-neutral facts:

engine.handle(new EventFacts(
    EventKind.EVASION,
    subjectUuid,
    subjectName,
    Map.of("score", posteriorProbability)
));

The producer is responsible for thread selection. DispatchEngine accepts calls from main and asynchronous event threads, subject to the tracker concurrency qualifications documented by stateful triggers.

Implementation notes, non-normative

The built-in event shape contains only the score-specific field:

"Evasion", new Event(
    "Evasion",
    List.of(
        new Field("subject", ACCOUNT),
        new Field("subject_name", TEXT),
        new Field("score", REAL)
    )
)

The runtime enum recognizes the source spelling:

case "evasion" -> EVASION;

The simulator creates the fact map:

case EVASION -> {
    double score = 1.0d;

    try {
        score = Double.parseDouble(data);
    } catch (final NumberFormatException ignored) {
        // Retain the default score.
    }

    facts.put("score", score);
}

The current dispatch boot stage registers only these live event listeners:

pm().registerEvents(
    new ChatEventListener(engine),
    plugin
);
pm().registerEvents(
    new BreakEventListener(engine),
    plugin
);
pm().registerEvents(
    new LoginEventListener(engine),
    plugin
);
  • No EvasionEventListener appears in the supplied boot sequence.
  • FingerprintService.raiseAlert sends localized staff messages but does not invoke the dispatch engine.
  • The Javadoc on EventKind.EVASION mentions fields score and banned, but the actual ALAM built-in event shape exposes score only. The language contract follows Builtins.EVENTS.
  • Attempting event.banned therefore produces E015.

Example

rule<Evasion> high_confidence_evasion {
    condition {
        trigger = Trigger::Atomic;
        guard = (event) ->
            event.score >= 0.95;
    }
    => verdict = (event) -> Sanction {
        .target = event.subject,
        .cite = { P.5.1 },
        .measure = Measure::CUSTODY,
        .duration = Duration::Permanent,
        .escalation = Escalation::None,
        .annotations = {
            Annotation::Evidence,
            Annotation::Notify("staff"),
            Annotation::Reason(
                "High-confidence evasion assessment"
            ),
        },
        .attribution = Attribution::System,
        .dry_run = DryRun::InheritSafety,
    };
}

Diagnostics

CodePhaseCondition

E015ElaborationA guard or verdict accesses a field not declared by Evasion, including banned.

The absence of a live event producer is not a compiler diagnostic. The rule remains valid and may be exercised through simulation or external integration.
