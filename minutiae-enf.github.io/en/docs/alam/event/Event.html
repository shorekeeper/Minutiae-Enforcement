Event

event

The distinguished base event shape containing the account identity and display name common to every concrete ALAM event.

Syntax

Event                                                     (1)

event.subject                                              (2)

event.subject_name                                         (3)
  • (1)The base event type used by event-polymorphic function positions.
  • (2)The stable account identity of the event subject.
  • (3)The display name captured with the event facts.

Fields

FieldTypePresenceDescription
subject Account Every event Stable UUID-backed identity of the player or account associated with the event.
subject_name Text Every event Display name observed when the event facts were created.

Constraints

  • Event is an internal base type, not a concrete event kind accepted by rule<...>.
  • A concrete event is assignable to Event.
  • Event is not assignable to a concrete event type.
  • A body checked against Event may access only subject and subject_name.
  • Concrete event types are nominal and are not assignable to one another.
  • An event value cannot be constructed directly by ALAM source.

Semantics

The base event type provides the common parameter type used by event-polymorphic templates. A template checked against Event can operate on any concrete event without knowing its event-specific fields.

Chat     assignable to Event
Break    assignable to Event
Login    assignable to Event
Evasion  assignable to Event

Event    not assignable to Chat, Break, Login, or Evasion

subject

The subject denotes an account identity rather than a display name. The interpreter stores canonical UUID text under the nominal Account type. Query built-ins decode that text to java.util.UUID at the environment boundary.

When no subject identity is available, event adaptation supplies an empty text representation. Query built-ins decode it as an unknown account and return their neutral result.

subject_name

The subject name is ordinary text captured with the event. It is not resolved again when the guard or verdict executes. A player rename after event creation cannot change the captured value.

The field is intended for display and pure descriptor construction. It is not a stable identity key and should not replace subject in enforcement queries.

Fact adaptation

Platform listeners and simulation construct EventFacts. Before interpretation, EventAdapter walks the lowered event record shape and creates one ALAM value per field.

Failure posture

A missing or incompatible event fact is adapted to a neutral value rather than producing an interpreter exception. Definitions should write guards whose comparisons fail safely for neutral input.

Event-polymorphic templates

A primary guard or verdict template body is checked against a function over the base Event type. It can therefore access common fields and be instantiated from rules over different concrete events.

Concrete-event fields such as message, block, ip, or score are unavailable in such a generic body unless the surrounding position supplies a concrete event function type.

Implementation notes, non-normative

The base event is declared explicitly:

public static final Event ANY_EVENT =
    new Event(
        "Event",
        List.of(
            new Field("subject", ACCOUNT),
            new Field("subject_name", TEXT)
        )
    );

Concrete events repeat the common fields before adding their own fields. The type model does not store a Java superclass pointer. Assignability recognizes the base event by name:

if (from instanceof Event
        && to instanceof Event target) {
    return target.name()
        .equals(Builtins.ANY_EVENT_NAME);
}

The adapter handles common fields separately from the raw fact map:

fields.add(switch (field.name()) {
    case "subject" ->
        new Value.TextV(
            facts.subject() == null
                ? ""
                : facts.subject().toString()
        );

    case "subject_name" ->
        new Value.TextV(
            facts.subjectName() == null
                ? ""
                : facts.subjectName()
        );

    default ->
        convert(
            facts.field(field.name()),
            field.type()
        );
});
  • EventFacts contains no Paper or Bukkit object.
  • Live listeners and /alam simulate feed the same plan and interpreter structures.
  • Lowering interns one Type.Rec shape per concrete event name inside a unit.
  • The record value allocates one field list per adapted event.

Example

template<Measure M>
verdict for_subject = (event) -> Sanction {
    .target = event.subject,
    .cite = { P.1.1 },
    .measure = M,
    .duration = Duration::Permanent,
    .escalation = Escalation::None,
    .annotations = {
        Annotation::Reason(event.subject_name),
    },
    .attribution = Attribution::System,
    .dry_run = DryRun::InheritSafety,
};
