Break

event

An observed successful player block-break event containing the canonical Bukkit material name of the broken block.

CategoryConcrete event
BaseEvent

Syntax

rule<Break> name { ... }                                 (1)

Step<Break> {
    .guard = break_guard,
}                                                          (2)
  • (1)Declares a rule whose event argument has type Break.
  • (2)Declares one block-break step of a sequence.

Fields

FieldTypeSourceDescription
subject Account Breaking player's UUID Stable identity of the player who broke the block.
subject_name Text Breaking player's current name Display name captured at event dispatch.
block Text event.getBlock().getType().name() Canonical uppercase Bukkit material enum name.

Constraints

  • Break must be spelled with its exact event type name in ALAM source.
  • An atomic or repeated break rule declares exactly one event.
  • A Step<Break> guard accepts one Break value and returns Bool.
  • The listener ignores a break event already cancelled by another plugin.
  • The block field is text. It is not a dedicated material enum in the ALAM type system.

Semantics

The listener observes a non-cancelled BlockBreakEvent at monitor priority and sends one immutable fact map to the dispatch engine. It never cancels the break event and never changes block drops, experience, or replacement state.

block

The block value is the Java enum constant name returned by Bukkit's material type. Current names are uppercase ASCII words separated by underscores, such as "STONE", "DIAMOND_ORE", or "OAK_LOG".

Definitions should compare against canonical material names. The listener performs no case normalization beyond the enum's own name() result.

Observation timing

The listener runs after ordinary lower-priority handlers. It observes the block type still available through the event's block reference. By Paper convention, a monitor handler is observational and does not alter the event.

Behavioral cancellation

The plugin separately registers an interaction listener at LOWEST priority. A ghosted or shadowed player's break event is cancelled there. Because the ALAM listener ignores cancelled events, that concealed player's attempted break does not reach ALAM dispatch.

Threading

Block-break dispatch normally begins on the Paper main thread. Guard evaluation must still avoid blocking I/O because the same interpreter and environment contracts are shared with asynchronous event kinds.

Implementation notes, non-normative

The built-in shape is:

"Break", new Event(
    "Break",
    List.of(
        new Field("subject", ACCOUNT),
        new Field("subject_name", TEXT),
        new Field("block", TEXT)
    )
)

The live adapter creates one field:

@EventHandler(
    priority = EventPriority.MONITOR,
    ignoreCancelled = true
)
public void onBreak(final BlockBreakEvent event) {
    engine.handle(new EventFacts(
        EventKind.BREAK,
        event.getPlayer().getUniqueId(),
        event.getPlayer().getName(),
        Map.of(
            "block",
            event.getBlock()
                .getType()
                .name()
        )
    ));
}
  • The fact map contains no world, coordinates, block state, tool, drop list, or cancellation cause.
  • ALAM cannot distinguish two worlds or locations from the current Break shape.
  • Material names follow the Paper server version and may gain new enum constants in later releases.
  • Simulation uppercases the supplied block text, approximating the live enum-name form.

Example

rule<Break> repeated_container_break {
    condition {
        trigger = Trigger::Repeated {
            .count = 4,
            .within = 2m,
            .group_by = GroupBy::Subject,
        };
        guard = (event) ->
            event.block == "CHEST"
            || event.block == "BARREL";
    }
    => verdict =
        griefing_verdict<Measure::SUSPENSION, 7d>;
}
