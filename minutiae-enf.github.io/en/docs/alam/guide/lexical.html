Lexical structure

ALAM is tokenized by a single left-to-right scan with maximal munch, bounded lookahead, and no regular expressions or backtracking in the Java lexer.

Tokens

token = keyword | identifier | wildcard | integer | real
      | duration | text | rule-identifier
      | punctuation | operator ;

identifier = identifier-start, { identifier-part } ;
identifier-start = letter | "_" ;
identifier-part = letter | digit | "_" ;

wildcard = "_" ;

An identifier that exactly matches a keyword is emitted as that keyword. A single underscore is the wildcard. Longer names containing underscores remain identifiers.

Literals

integer A non-empty decimal digit sequence decoded to a signed 64-bit value during lexing.
real Digits, one decimal point, and at least one digit after the point. A leading sign is a separate unary operator.
duration One or more alternating digit and unit segments. Units are s, m, h, d, and w.
text A double-quoted string. Supported escapes are \", \\, \n, \t, and \r.
rule identifier The letter P followed immediately by one or more dot-decimal segments, for example P.2.1.

Maximal-munch refinements

A digit run followed immediately by a duration unit begins one duration token. The complete span must preserve strict digit-unit alternation. Therefore 3hx is a malformed duration, not a valid 3h followed by an identifier.

The word P begins a rule identifier only when the next two characters are a dot and a digit. Otherwise it remains an ordinary identifier.

Whitespace and comments

line-comment  = "//", { character }, newline ;
block-comment = "/*", { character }, "*/" ;

Whitespace and comments are discarded between tokens. Block comments do not nest. An unterminated block comment consumes the remaining source and produces L006.

Operators and punctuation

{ } ( ) [ ] , ;
:: : -> => = == != < > <= >=
&& || ! .. . + - * / ~

A lone ampersand produces L007; a lone vertical bar produces L008. The lexer does not reinterpret them as partial boolean operators.

Recovery under the Java implementation

Tokens are stored in parallel primitive arrays inside TokenBuffer. Each slot carries the token kind ordinal, source offsets, line, column, numeric payload, and optional string-pool index. Identifiers and keywords are not allocated during classification; their text is materialized from the retained source only when a later phase requests it.

A lexical error emits an ERROR token and scanning continues. The stream always terminates with an EOF token, so the parser can retain valid sibling declarations after a malformed span.

Example

const Duration warning_window = 1h30m;

rule<Chat> long_message {
    condition {
        trigger = Trigger::Atomic;
        guard = (event) -> event.length > 120;
    }
    => verdict = make_warning<P.2.1>;
}
