Compilation pipeline

An ALAM file is compiled completely before any rule from that file is armed. A phase error rejects the unit as a whole and never publishes a partial plan.

Phase gating

Lexing, parsing, and elaboration share one diagnostic collection. Parsing and elaboration retain valid sibling declarations to provide more than one useful diagnostic per run. Evaluation starts only if no earlier error exists. Verification starts only if evaluation succeeds. Lowering starts only after verification reports no error.

  -> declarations

  -> residual values
  -> verified artifacts

Elaboration

The elaborator processes declarations in source order. Names become visible only after declaration, which makes forward references and definition cycles unrepresentable. It performs bidirectional type checking, effect checking, member resolution, constructor resolution, match exhaustiveness, and automaton item validation.

Instead of building a second tree, Java stores inferred types and resolved symbols in IdentityHashMap side tables keyed by AST node identity. Evaluation walks the original syntax and consults these tables.

Compile-time evaluation

Constants, matrices, transforms, template instantiations, expansions, and automaton set algebra are normalized at load time. Guard and verdict bodies are captured as closures rather than executed. Each closure retains its lexical Env chain, so constants and template parameters are already available when dispatch evaluates it.

Arithmetic is checked. Integer overflow, division by zero, negative duration results, invalid text patterns, and evaluation-depth exhaustion become diagnostics rather than uncaught Java exceptions.

Verification

Verification checks facts not fully represented by static types: rule identifiers must exist in the external rule registry, layout measure and duration terms must agree, escalation ladders must be valid, layout keys must be non-blank, and stamped layouts may carry only configuration-scoped annotations.

Lowering

Lowering is a structural translation with no user-facing failure mode. Trigger values become primitive TriggerPlan records, event names become EventKind values, duration terms become millisecond spans, and partition selectors become TriggerPlan.Part values.

File loading and publication

AlamService scans <data>/alam/*.alam in lexical file-name order. A failed file is skipped, while accepted sibling files remain available. Accepted plans are copied into immutable lists and published through volatile references.

Stamped layout collisions are deterministic. The first accepted producer in file-name order owns the key; later producers are logged and dropped. File-authored entries in layouts.yml remain authoritative over stamped definitions.

Example

const Duration span = 30s;

rule<Chat> flood {
    condition {
        trigger = Trigger::Repeated(5, span, GroupBy::Subject);
        guard = (event) -> event.length > 120;
    }
    => verdict = flood_verdict<Measure::MUTE, 3h>;
}

automaton chat_watch {
    use flood;
}
