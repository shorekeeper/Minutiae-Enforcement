MEMinutiae Enforcement docs

Runtime model

The runtime consumes immutable plans. It adapts platform events, advances trigger state, evaluates guards and verdicts, and routes the resulting sanction through the same resolver and executor used by manual commands.

Event indexing

At construction, DispatchEngine indexes each rule under every event kind observed by its trigger. An atomic or repeated rule has one index entry. A sequence rule is indexed under every step kind, allowing any participating event to advance its partial match.

Event facts

Listeners translate Paper events into EventFacts, a platform-neutral record containing the event kind, subject UUID, display name, and a flat map of typed fields. EventAdapter converts those facts into the nominal ALAM event record expected by the plan.

An absent or incompatible fact becomes the field type's neutral value: zero, false, or empty text. Malformed platform input therefore tends to fail guard comparisons instead of faulting dispatch.

Gating order

  1. Reject a rule whose automaton is self-muted.
  2. Apply the trigger-specific state transition.
  3. Apply the rule guard to the event argument vector.
  4. Acquire one firing from the automaton's per-minute throttle.
  5. Evaluate the verdict and materialize its sanction descriptor.

For atomic and repeated triggers, the guard is evaluated before a repeated occurrence enters its window. For a sequence, step guards decide admission while the rule guard runs only after completion against the full captured argument vector.

Partitions

GroupBy::SubjectOne state partition per subject UUID.
GroupBy::GlobalOne state partition shared by all events.
GroupBy::FieldOne state partition per rendered value of a named event field.

The internal key combines automaton name, rule name, and rendered partition. State is therefore isolated between rules even when their partition values are equal.

Window and sequence tracking

Trigger::Repeated uses a fixed-capacity primitive timestamp ring per partition. A firing clears that ring, so another firing must accrue a new full count.

Trigger::Sequence keeps one partial match per partition. An event that matches the first step while another step is expected re-anchors the sequence at that event. Admitted step facts are captured in order. On completion, the capture list is drained and converted into one argument per declared event.

Throttle and self-muting

Each automaton owns a fixed-capacity timestamp ring for its one-minute firing limit. Saturating the ring rejects the firing and adds the automaton to the self-mute set. Subsequent events are rejected before trigger state is consulted, bounding the blast radius of a defective definition.

Verdict materialization

The verdict returns a total Sanction descriptor. SanctionMaterializer translates it into the manual command surface: citations become count directives, the measure becomes a measure directive, a fixed duration becomes a duration override, and annotation variants become annotation tokens.

The resulting command is passed to SanctionResolver under the configured system actor's permission predicate. Automatic enforcement cannot exceed the actor's grant because it does not bypass manual validation.

Arming and dry-run

An automaton not present in the armed set is evaluated in forced dry-run. The plan, guard, verdict, materializer, and resolver still run, but the executor persists and applies nothing. A descriptor may independently force dry-run through DryRun::Forced.

Threading

Dispatch may run on the main server thread or the asynchronous chat thread. Tracking maps are concurrent. The interpreter is reentrant. Query built-ins perform cached reads only. Storage and player-facing effects are delegated to existing asynchronous and main-thread boundaries in the sanction executor.

Simulation

The /alam simulate path does not consume window, sequence, throttle, capture, or mute state. Repeated triggers are treated as if their recurrence condition were satisfied after the guard. Sequence simulation tests the final step and synthesizes earlier argument positions from the simulated event because no live partial state is consulted.
