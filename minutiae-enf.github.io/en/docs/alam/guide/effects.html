Effects

ALAM separates deterministic value construction from read-only access to the live enforcement environment through a two-point effect system.

Effect order

pure < query
pure Arithmetic, comparison, matching, construction, field access, compile-time transformation, and calls to pure functions.
query Everything permitted by pure, plus calls that read the current enforcement environment.

An operation of effect E may be called only where the position grants an allowance A and E <= A.

Position allowances

PositionAllowance
const initializerpure
matrix cellpure

Guard bodyquery
Sequence step guard bodyquery
Verdict bodypure

Function effects

A function type carries the effect granted to its body. Function assignability is contravariant in parameters, covariant in the result, and monotonic in effect. A pure function may be used where a query-capable function is accepted, but a query-capable function cannot be placed in a pure function position.

Query built-ins

precedentReads a cached prior-sanction count.
fingerprint_scoreReads a cached evasion score.
is_onlineReads live online presence.
nowReads the current epoch-millisecond timestamp.

All four are query. None performs blocking storage I/O directly. Cache misses for storage-backed values return a neutral value and schedule an asynchronous fill.

Example

rule<Login> repeat_evasion {
    condition {
        trigger = Trigger::Atomic;
        guard = (event) ->
            fingerprint_score(event.subject) >= 0.90
            && precedent(event.subject, P.5.1) > 0;
    }
    => verdict = evasion_verdict<Measure::CUSTODY>;
}

The guard may call fingerprint_score and precedent because its body is elaborated under query. Calling either from a verdict or constant initializer produces E007.

Runtime failure posture

A runtime query or expression failure does not escape into a platform event handler. A guard failure is treated as an unsatisfied guard. A verdict failure suppresses the firing and writes an English operator diagnostic.
