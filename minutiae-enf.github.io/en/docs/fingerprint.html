Evasion detection

When a banned player returns on a fresh account, the connection itself betrays them. The fingerprint subsystem scores every login against the recorded signals of banned accounts and alerts staff when the match is strong. It never bans anyone on its own.

How it works, briefly

  1. When a ban is issued, the target's connection signals are captured and stored with the sanction: address, network prefix, client locale and brand, name pattern, client settings.
  2. On every login, a probe of the incoming connection is compared against those stored signals.
  3. Each agreeing signal contributes evidence weighted by how rare the shared value is on your server: an address shared by two accounts means a lot; a locale shared by half the player base means almost nothing. Evidence decays with the age of the stored observation.
  4. The combined evidence yields a probability. At or above the flag threshold, holders of minutiae.fingerprint.alert are notified.
Note

Alerts only, by design. The model is calibrated and explains itself, but shared households, siblings, and VPNs exist - a human decides. Automation on top of evasion scores is possible through the ALAM engine, where it is subject to its own safety posture.

The signals

ip-full The exact address. The strongest signal, and the fastest to decay (half-life 14 days) - addresses churn.
ip-subnet The /24 or /64 prefix. Survives address churn within a provider; weaker but slower to decay.
ip-asn The provider label, when the address matches your configured network catalogue.
ip-rdns Reverse-DNS host pattern, digits normalised. Off by default (costs a lookup per fresh connection).
name-pattern The player name with digits collapsed: Steve123 and Steve777 match.
locale, client-brand, client-protocol, view-distance, skin-parts, main-hand Client fingerprint: individually weak, collectively meaningful.

Signals derived from one attribute do not double-count: a full address match already implies the subnet and provider, so only the most specific one contributes. A subnet match without a full-address match still counts on its own - that is exactly the changed-address case the subnet signal exists for.

Tuning

fingerprint:
  enabled: true
  reverse-dns: false
  evidence:
    prior-evasion-rate: 0.001
    flag-threshold: 0.90
flag-threshold The posterior probability that raises an alert. Lower it toward 0.8 if evaders slip through; raise it toward 0.95 if staff report noise. Change one knob at a time and give it a few days.
prior-evasion-rate The assumed base rate of evasion among connections. The default is deliberately conservative; increase (e.g. 0.01) only if your server is under active ban-evasion pressure.
reverse-dns Adds the PTR-record host pattern as a signal. Useful against mobile-carrier address churn; costs one DNS lookup per fresh connection, bounded and off the main thread.

The remaining evidence, correlation, and clustering knobs are documented inline in config.yml; the defaults are sensible and rarely need touching.

The network catalogue

fingerprint:
  network:
    entries:
      - { cidr: "203.0.113.0/24", asn: "EXAMPLE-VPN", datacenter: true }
      - { cidr: "198.51.100.0/24", asn: "EXAMPLE-ISP", datacenter: false }

Labels address ranges with a provider name and a datacenter/VPN flag. You do not need to blacklist VPNs: a VPN exit shared by many accounts automatically carries near-zero evidence because the model weighs values by how many distinct accounts bear them. The catalogue just makes the labelling visible in the panel.

Behavioural correlation

Field agreement cannot tell an alt account from a sibling on the same connection. Temporal behaviour helps:

Handoffs One account logs out, another logs in from the same address within minutes - the alternating pattern of a single operator. Raises the score.
Simultaneous presence Both accounts online at once - characteristic of two people. Lowers the score.
Active hours Similar hour-of-day patterns weakly raise it.

Every term is capped and the total adjustment is bounded, so behaviour can tip a borderline case but never override the field evidence.

Inspecting a suspect

In game:

/me-ban fingerprint SomePlayer

Shows the live session's signals and the best match with its score. For the full picture, the panel's HKEY_FINGERPRINTS hive offers three views:

ByType Every recorded signal value and the accounts bearing it - "who else was on this address".
Clusters Accounts grouped by shared rare signals. Rare values merge accounts outright; common ones only accumulate evidence toward a merge; near-universal ones are discarded, so a popular VPN cannot fuse your player base into one blob.
Suspects / Evidence For an online player: the signal-by-signal breakdown of their score - each signal's contribution, the rarity estimate behind it, and its age decay. This is the view to screenshot into a staff dispute.

Teaching the model

Signal reliabilities are not fixed: when staff adjudicate a flag as confirmed evasion or a false positive, the weights of the signal types that fired update accordingly and persist. A server where locale turns out to be noise will gradually trust locale less. No configuration is involved; the model learns from verdicts.

Data and privacy

  • Signals are captured only when a connection-blocking sanction is issued - not for the general population.
  • Everything stays in your local SQLite file; nothing is sent anywhere.
  • Addresses are visible to panel users and in signal listings; scope your read tokens accordingly.
