Layouts and escalation

A layout encodes your punishment policy once: moderators type ::griefing, and the rule, measure, duration, ladder, and default annotations all come from configuration instead of memory.

Anatomy of a layout

layouts:
  griefing:
    rule: "P.3.2"
    measure: SUSPENSION
    reason: "Griefing of player-owned structures"
    duration: 7d
    escalation: [ "7d", "30d", "permanent" ]
    annotations:
      - "@evidence(required)"
      - "@escalate"
      - "@notify(staff)"
rule The cited rule identifier. Mandatory for any invocable layout; a layout naming a rule absent from rules.yml is disabled at load with a logged error, never silently.
measure The enforcement mechanism. Mandatory; see Measures for the temporal rules each measure imposes.
reason Display reason. When omitted, the rule's description from rules.yml is used.
duration Base duration. Required for temporal measures (MUTE, QUARANTINE, SUSPENSION); forbidden for instantaneous ones (WARN, CENSURE, KICK); ignored by CUSTODY, which is permanent by definition.
escalation The ladder consulted when @escalate is present. Covered below.
annotations Default annotations, in the same grammar as commands. Only config-scoped annotations are allowed here; inline-only directives are rejected at load.
Note

Layout annotations are permission-checked against the issuing moderator, not against the configuration. A layout carrying @shadow is unusable by staff whose group lacks minutiae.annotation.shadow - the command is rejected, not silently stripped.

Inheritance

A key starting with an underscore is private: it never appears in commands or tab completion and exists only as a base. A layout names its parent with extends:

layouts:
  _base_cheat:
    rule: "P.5.1"
    annotations:
      - "@evidence(required)"
      - "@ip-lock"

  cheating:
    extends: _base_cheat
    measure: CUSTODY
    reason: "Use of unauthorised client modifications"

  killaura:
    extends: _base_cheat
    measure: CUSTODY
    reason: "KillAura"
    annotations:
      - "@shadow"

Merge rules are simple and worth knowing exactly:

scalars rule, reason, measure, duration: the child value wins when present, otherwise the parent's is inherited.
escalation Replaced wholesale by the child when declared; ladders never merge rung by rung.
annotations Concatenated: parent's first, then the child's. They accumulate down the chain.

Inheritance cycles, missing parents, and children of failed parents are each detected at load; the offending layout is disabled and the rest of the registry loads normally. /me-ban layouts shows what actually made it in.

Escalation ladders

With @escalate present, the duration is not taken from duration but from the ladder, indexed by the player's precedent - the count of their prior non-warning sanctions under the same rule:

escalation: [ "7d", "30d", "permanent" ]

first offence   -> 7d
second offence  -> 30d
third and later -> permanent   (the last rung repeats)
  • Only the final rung may be permanent.
  • An explicit duration=... override on the command bypasses the ladder entirely for that sanction.
  • @tariff(1d) applies a minimum floor after the ladder picks a rung.

What counts as precedent

Precedent is per rule, and three mechanisms modify it:

Warnings Counted separately: they feed the @warn-first gate, not the ladder.
Lift kind A vacated sanction leaves precedent; a pardoned or time-served one stays. See Lifting sanctions.
Decay @decay(30d) ages a sanction out of precedent after the window, without lifting it.

Probation

@probation(2w) - at issue time or on a pardon/served lift - opens a window during which a repeat offence under the rule is aggravated: one extra ladder rung, or a duration multiplier, per the escalation policy.

The warn-first gate

toxic:
  rule: "P.2.4"
  measure: MUTE
  duration: 3h
  annotations:
    - "@warn-first(2)"

The first two offences under P.2.4 become formal warnings; the third applies the mute. The gate is bypassed per command with @now, for cases that do not deserve the courtesy.

Joined cases

@weave(#42) on a command attaches the new sanction under an existing one. Chain members share precedent - the ladder consults the whole chain, not just the rule-scoped history - and a cascade lift can close the entire case at once. Use it when one incident produces several sanctions that should be judged as one course of conduct.

Overriding a layout per command

The layout is a starting point, not a straitjacket:

# Change the measure (directional permission applies):
/enforce Steve ::cheating @commute(SUSPENSION) duration=30d

# Remove a layout-supplied annotation for this one sanction:
/enforce Steve ::killaura !@shadow

# Override duration and reason:
/enforce Steve ::griefing duration=14d reason="Second base this week"

@measure is rejected when a layout is present - changing a layout's measure is always @commute, so the mitigate/aggravate permission split applies.

Editing workflow

  1. Edit layouts.yml.
  2. /me-ban reload - layouts reload live, revalidated against rules and annotations.
  3. Check the log line: registry ready: N invocable, M private, K disabled. A non-zero disabled means an error line above names the offender.
  4. Inspect the result: /me-ban layouts <key>.
  5. Rehearse: /enforce TestAccount ::key @dry-run.
