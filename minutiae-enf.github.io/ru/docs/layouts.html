Шаблоны и эскалация

Шаблон записывает вашу политику наказаний один раз: модератор набирает ::griefing, а правило, мера, срок, лестница и аннотации по умолчанию приходят из конфигурации, а не из памяти.

Устройство шаблона

layouts:
  griefing:
    rule: "P.3.2"
    measure: SUSPENSION
    reason: "Гриферство построек игроков"
    duration: 7d
    escalation: [ "7d", "30d", "permanent" ]
    annotations:
      - "@evidence(required)"
      - "@escalate"
      - "@notify(staff)"
rule Идентификатор цитируемого правила. Обязателен для любого вызываемого шаблона; шаблон с правилом, отсутствующим в rules.yml, отключается при загрузке с ошибкой в логе — никогда молча.
measure Механизм принуждения. Обязателен; временные правила каждой меры — на странице Меры.
reason Отображаемая причина. При отсутствии берётся описание правила из rules.yml.
duration Базовый срок. Обязателен для временных мер (MUTE, QUARANTINE, SUSPENSION); запрещён для мгновенных (WARN, CENSURE, KICK); игнорируется CUSTODY, которая перманентна по определению.
escalation Лестница, к которой обращается @escalate. Разобрана ниже.
annotations Аннотации по умолчанию, в той же грамматике, что и команды. Здесь допустимы только config-scoped аннотации; директивы только-для-команды отклоняются при загрузке.
Примечание

Аннотации шаблона проверяются по правам выдающего модератора, а не конфигурации. Шаблон с @shadow непригоден для персонала, чья группа лишена minutiae.annotation.shadow, — команда отклоняется, а не молча урезается.

Наследование

Ключ, начинающийся с подчёркивания, — приватный: он никогда не появляется в командах и автодополнении и существует только как база. Шаблон называет родителя через extends:

layouts:
  _base_cheat:
    rule: "P.5.1"
    annotations:
      - "@evidence(required)"
      - "@ip-lock"

  cheating:
    extends: _base_cheat
    measure: CUSTODY
    reason: "Использование неразрешённых модификаций клиента"

  killaura:
    extends: _base_cheat
    measure: CUSTODY
    reason: "KillAura"
    annotations:
      - "@shadow"

Правила слияния просты, и их стоит знать точно:

скаляры rule, reason, measure, duration: значение потомка побеждает, когда присутствует, иначе наследуется родительское.
escalation Заменяется потомком целиком, когда объявлена; лестницы не сливаются поступенчато.
annotations Конкатенируются: сначала родительские, затем потомка. Накапливаются вниз по цепочке.

Циклы наследования, отсутствующие родители и потомки павших родителей обнаруживаются при загрузке; провинившийся шаблон отключается, остальной реестр загружается нормально. /me-ban layouts показывает, что фактически попало внутрь.

Лестницы эскалации

С @escalate срок берётся не из duration, а из лестницы, индексируемой прецедентом игрока — счётом его прежних непредупредительных санкций по тому же правилу:

escalation: [ "7d", "30d", "permanent" ]

первое нарушение  -> 7d
второе нарушение  -> 30d
третье и далее    -> permanent   (последняя ступень повторяется)
  • Только последняя ступень может быть permanent.
  • Явное переопределение duration=... на команде полностью обходит лестницу для этой санкции.
  • @tariff(1d) применяет нижний порог после выбора ступени лестницей.

Что считается прецедентом

Прецедент ведётся по правилу, и три механизма его меняют:

Предупреждения Считаются отдельно: они питают ворота @warn-first, а не лестницу.
Вид снятия Аннулированная санкция выходит из прецедента; помилованная или отбытая остаётся. См. Снятие санкций.
Затухание @decay(30d) выводит санкцию из прецедента по истечении окна, не снимая её.

Пробация

@probation(2w) — при выдаче или при снятии pardon/served — открывает окно, в течение которого рецидив по правилу отягчается: одна лишняя ступень лестницы или множитель срока, согласно политике эскалации.

Ворота warn-first

toxic:
  rule: "P.2.4"
  measure: MUTE
  duration: 3h
  annotations:
    - "@warn-first(2)"

Первые два нарушения по P.2.4 становятся формальными предупреждениями; третье применяет мут. Ворота обходятся на одну команду через @now — для случаев, не заслуживающих такой любезности.

Объединённые дела

@weave(#42) на команде подшивает новую санкцию под существующую. Участники цепочки разделяют прецедент — лестница смотрит на всю цепочку, а не только на историю по правилу, — и каскадное снятие может закрыть всё дело разом. Применяйте, когда один инцидент порождает несколько санкций, которые должны судиться как одна линия поведения.

Переопределение шаблона на команде

Шаблон — отправная точка, не смирительная рубашка:

# Сменить меру (действует направленное право):
/enforce Steve ::cheating @commute(SUSPENSION) duration=30d

# Убрать аннотацию шаблона для этой одной санкции:
/enforce Steve ::killaura !@shadow

# Переопределить срок и причину:
/enforce Steve ::griefing duration=14d reason="Вторая база за неделю"

@measure отклоняется при наличии шаблона — смена меры шаблона всегда идёт через @commute, так что действует разделение прав на смягчение и отягчение.

Порядок правки

  1. Правьте layouts.yml.
  2. /me-ban reload — шаблоны перезагружаются вживую с ревалидацией по правилам и аннотациям.
  3. Проверьте строку лога: registry ready: N invocable, M private, K disabled. Ненулевой disabled означает, что строка ошибки выше называет виновника.
  4. Осмотрите результат: /me-ban layouts <key>.
  5. Отрепетируйте: /enforce TestAccount ::key @dry-run.
